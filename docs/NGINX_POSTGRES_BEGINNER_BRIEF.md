# Памятка начинающему backend-разработчику по nginx и PostgreSQL

Этот документ нужен, чтобы подготовиться к вопросам по задаче: сотрудники отдела должны подготовить конфигурационные файлы nginx и PostgreSQL.

Ты не обязан сразу знать все тонкости администрирования. Нормальная позиция начинающего backend-разработчика: понимать назначение компонентов, знать какие вводные нужны, уметь задавать правильные уточняющие вопросы и не принимать опасные настройки “на веру”.

## 1. Коротко: что вообще происходит

В типичном backend-приложении есть несколько частей:

- пользователь открывает сайт или отправляет API-запрос;
- запрос приходит на сервер;
- nginx принимает внешний HTTP/HTTPS-трафик;
- nginx передает запрос backend-приложению;
- backend-приложение работает с PostgreSQL;
- PostgreSQL хранит данные.

Примерная схема:

```text
Пользователь
    |
    | HTTPS, порт 443
    v
nginx
    |
    | внутренний HTTP, например 127.0.0.1:8000
    v
backend-приложение
    |
    | PostgreSQL protocol, порт 5432
    v
PostgreSQL
```

nginx и PostgreSQL решают разные задачи:

- nginx отвечает за входящие веб-запросы, HTTPS, проксирование, статику, базовые лимиты и веб-логи;
- PostgreSQL отвечает за хранение данных, подключения к базе, безопасность доступа, производительность, логи запросов и возможность восстановления данных.

## 2. Твоя роль в этой задаче

Твоя задача не обязательно написать идеальный production-конфиг самому. Скорее всего, твоя роль такая:

- понять, какие конфиги сотрудники должны подготовить;
- проверить, что они не сделали очевидно опасных вещей;
- собрать недостающие вопросы;
- передать вопросы ответственным людям: DevOps, системному администратору, архитектору, тимлиду, специалисту по ИБ;
- принять результат с понятной инструкцией запуска и проверки.

Хорошая рабочая фраза:

> Я не хочу принимать универсальный конфиг без контекста. Давайте зафиксируем окружение, схему сети, домены, порты, доступы, логи, бэкапы и команды проверки.

## 3. Что такое nginx простыми словами

nginx - это веб-сервер и reverse proxy.

В роли веб-сервера он может отдавать файлы:

- HTML;
- CSS;
- JS;
- изображения;
- загруженные пользователями файлы, если так спроектировано.

В роли reverse proxy он принимает запрос снаружи и пересылает его внутрь приложения.

Например:

```text
https://example.ru/api/users
```

приходит в nginx, а nginx пересылает его backend-сервису:

```text
http://127.0.0.1:8000/api/users
```

Пользователь не видит внутренний адрес backend-приложения. Он видит только домен и HTTPS.

## 4. Что обычно должно быть в конфиге nginx

Обычно для проекта нужен не один огромный глобальный `nginx.conf`, а отдельный конфиг сайта или сервиса. Он может называться, например:

- `app.conf`;
- `site.conf`;
- `default.conf`;
- `nginx.conf`, если это Docker-контейнер с nginx.

В конфиге nginx обычно указывают:

- на каких портах слушать запросы: `80`, `443`;
- какой домен обслуживать: `server_name example.ru`;
- куда проксировать backend: `proxy_pass http://127.0.0.1:8000`;
- где лежит frontend-статика, если nginx ее отдает;
- где сертификаты HTTPS;
- нужно ли перенаправлять HTTP на HTTPS;
- какие заголовки передавать backend-приложению;
- лимиты размера запроса;
- таймауты;
- пути к логам.

## 5. Частые вопросы по nginx и как на них отвечать

### Вопрос: какой домен у приложения?

Что хотят узнать:

- какой адрес будет у сервиса;
- что писать в `server_name`;
- для какого домена выпускать сертификат.

Если ты знаешь домен:

> Домен приложения: `example.ru`. Для тестового окружения используется `test.example.ru`.

Если не знаешь:

> Домен еще не зафиксирован. В конфиге нужно оставить параметр или placeholder, а вопрос по домену вынести на согласование с ответственным за инфраструктуру.

Важно: не надо придумывать домен самому, если его не согласовали.

### Вопрос: нужен ли HTTPS?

Для production почти всегда да.

Безопасный ответ:

> Для production HTTPS обязателен. Нужно уточнить источник сертификатов: Let's Encrypt, корпоративный удостоверяющий центр или уже выданные сертификаты.

Для локальной разработки HTTPS может быть не нужен:

> Для локального окружения можно использовать HTTP, но для stage/prod нужно закладывать HTTPS.

### Вопрос: откуда брать SSL-сертификаты?

Сертификаты нужны для HTTPS.

Варианты:

- Let's Encrypt;
- корпоративный центр сертификации;
- сертификаты, которые уже выдала инфраструктурная команда;
- самоподписанные сертификаты только для dev/test, если это разрешено.

Ответ, если не знаешь:

> Источник сертификатов не указан. Нужно уточнить у инфраструктуры или ИБ. В конфиге нельзя оставлять случайные локальные сертификаты для production.

### Вопрос: куда nginx должен проксировать запросы?

Имеется в виду адрес backend-приложения.

Примеры:

```text
127.0.0.1:8000
app:8000
backend:8080
10.10.1.15:8000
```

Если приложение в Docker Compose, часто используется имя сервиса:

```nginx
proxy_pass http://backend:8000;
```

Если приложение на той же машине без Docker:

```nginx
proxy_pass http://127.0.0.1:8000;
```

Безопасный ответ:

> Нужен фактический внутренний адрес backend-сервиса для конкретного окружения. Для Docker это может быть имя сервиса, для обычной ВМ - localhost и порт приложения.

### Вопрос: nginx должен отдавать frontend или только API?

Варианты:

- nginx отдает собранный frontend и проксирует `/api` в backend;
- nginx только проксирует все запросы в backend;
- frontend находится отдельно, например в другом сервисе.

Ответ, если не знаешь:

> Нужно уточнить схему деплоя frontend. Если frontend собирается в статические файлы, nginx может отдавать `dist`/`build`, а `/api` проксировать в backend.

### Вопрос: какие нужны лимиты на размер запроса?

Например, пользователь загружает файл. Если лимит маленький, загрузка будет падать.

В nginx за это часто отвечает:

```nginx
client_max_body_size 20m;
```

Как отвечать:

> Лимит зависит от максимального размера файлов, которые разрешено загружать. Если загрузок нет, можно оставить небольшой лимит. Если есть загрузка документов, нужно согласовать максимальный размер файла.

### Вопрос: какие таймауты ставить?

Таймауты определяют, сколько nginx ждет ответ от backend.

Слишком маленький таймаут может оборвать долгий запрос. Слишком большой может скрывать проблемы.

Ответ:

> Таймауты нужно выбирать по реальному поведению API. Для обычных быстрых API запросы должны отвечать за секунды. Для долгих операций лучше использовать фоновые задачи, а не держать HTTP-запрос открытым.

### Вопрос: какие логи нужны в nginx?

Обычно есть:

- `access.log` - кто, когда и какой запрос сделал;
- `error.log` - ошибки nginx.

Ответ:

> Нужны access/error логи, путь хранения и ротация. Для production желательно согласовать формат логов с системой мониторинга или сбора логов.

## 6. Что обязательно проверить в nginx-конфиге

Попроси сотрудников показать:

- файл конфигурации;
- для какого окружения он предназначен;
- какие переменные нужно заменить;
- команду проверки;
- команду перезагрузки nginx;
- пример curl-запроса для проверки.

Минимальные команды проверки:

```bash
nginx -t
systemctl reload nginx
```

Если nginx в Docker:

```bash
docker compose exec nginx nginx -t
docker compose restart nginx
```

Важно: команды могут отличаться в конкретном проекте.

## 7. Красные флаги в nginx

Стоит насторожиться, если:

- production работает только по HTTP без HTTPS;
- в конфиге нет `server_name` или стоит случайный домен;
- все запросы проксируются на непонятный IP;
- нет `X-Forwarded-For`, `X-Forwarded-Proto`, `Host`;
- слишком большой `client_max_body_size` без причины, например `1000m`;
- отключена проверка TLS без объяснения;
- конфиг не проходит `nginx -t`;
- нет инструкции, куда положить файл;
- нет инструкции, как откатить изменения;
- в конфиге есть реальные секреты, токены, пароли.

## 8. Что такое PostgreSQL простыми словами

PostgreSQL - это база данных. Backend-приложение подключается к ней, чтобы читать и сохранять данные.

PostgreSQL обычно использует порт:

```text
5432
```

Для подключения нужны:

- host;
- port;
- database name;
- user;
- password;
- иногда SSL-настройки.

Пример строки подключения:

```text
postgresql://app_user:password@db:5432/app_db
```

Важно: пароль нельзя хранить в публичном конфиге или документации. Вместо него используют переменные окружения, secret-хранилище или защищенные настройки окружения.

## 9. Какие конфиги PostgreSQL обычно нужны

Чаще всего обсуждают два файла:

### `postgresql.conf`

Главный конфиг PostgreSQL.

Там настраиваются:

- адреса, на которых PostgreSQL слушает подключения;
- порт;
- лимиты подключений;
- использование памяти;
- логи;
- WAL;
- репликация;
- автovacuum;
- SSL;
- параметры производительности.

### `pg_hba.conf`

Файл правил доступа.

Он отвечает на вопросы:

- кто может подключаться;
- к какой базе;
- с какого IP-адреса или подсети;
- каким способом аутентификации.

Очень упрощенный пример:

```text
host    app_db    app_user    10.10.1.0/24    scram-sha-256
```

Это означает:

```text
Пользователь app_user может подключаться к базе app_db
с адресов подсети 10.10.1.0/24
с проверкой пароля через scram-sha-256.
```

## 10. Частые вопросы по PostgreSQL и как на них отвечать

### Вопрос: какая версия PostgreSQL?

Версия важна, потому что параметры могут отличаться.

Ответ, если знаешь:

> Используем PostgreSQL 15.

Если не знаешь:

> Версия PostgreSQL не зафиксирована. Нужно уточнить, потому что конфигурация и доступные параметры зависят от версии.

### Вопрос: где будет запущена база?

Варианты:

- на той же машине, что backend;
- на отдельной ВМ;
- в Docker;
- в Kubernetes;
- в managed-сервисе;
- на существующем сервере БД.

Ответ:

> Нужно подтвердить схему размещения базы. От этого зависят `listen_addresses`, правила `pg_hba.conf`, firewall, бэкапы и мониторинг.

### Вопрос: PostgreSQL должен быть доступен снаружи?

Правильная базовая позиция:

> Нет, PostgreSQL не должен быть открыт в интернет. Доступ должен быть только от backend-сервисов, администраторов и служебных систем из разрешенных сетей.

Если кто-то предлагает открыть `5432` наружу:

> Нужно отдельное согласование с ИБ/инфраструктурой. Лучше использовать VPN, bastion host или внутреннюю сеть.

### Вопрос: что писать в `listen_addresses`?

`listen_addresses` определяет, на каких сетевых интерфейсах PostgreSQL принимает подключения.

Примеры:

```conf
listen_addresses = 'localhost'
```

Только локальные подключения с той же машины.

```conf
listen_addresses = '*'
```

PostgreSQL слушает все интерфейсы. Это не всегда плохо, но опасно без строгого `pg_hba.conf` и firewall.

Безопасный ответ:

> Значение зависит от размещения backend и базы. Если backend на той же машине, достаточно `localhost`. Если backend на другом сервере, нужен конкретный внутренний интерфейс или адрес, плюс строгие правила `pg_hba.conf` и firewall.

### Вопрос: какой `max_connections` ставить?

`max_connections` - максимум одновременных подключений к PostgreSQL.

Важно: большое значение не всегда хорошо. Каждое подключение потребляет ресурсы.

Ответ:

> Значение зависит от приложения, пула соединений и нагрузки. Лучше использовать connection pool, например PgBouncer или пул в приложении, чем просто сильно увеличивать `max_connections`.

Если не знаешь:

> Пока нет данных по нагрузке и пулу соединений. Нужно заложить разумное стартовое значение и отдельно проверить, как приложение управляет подключениями.

### Вопрос: какие параметры памяти ставить?

В PostgreSQL часто обсуждают:

- `shared_buffers`;
- `work_mem`;
- `maintenance_work_mem`;
- `effective_cache_size`.

Для них нужно знать RAM сервера.

Ответ:

> Параметры памяти нельзя корректно выбрать без информации о RAM сервера, типе нагрузки и том, какие еще процессы работают на машине.

Главное: не соглашайся на “универсальные оптимальные параметры” без ресурсов сервера.

### Вопрос: нужно ли логировать медленные запросы?

Да, для production это полезно.

Часто используют:

```conf
log_min_duration_statement = 1000
```

Это означает: логировать запросы дольше 1000 мс.

Ответ:

> Для production желательно включить логирование медленных запросов. Порог нужно согласовать, например начать с 1000 мс и корректировать после наблюдения.

### Вопрос: какая аутентификация нужна?

Сейчас хороший базовый вариант для паролей:

```conf
password_encryption = scram-sha-256
```

В `pg_hba.conf`:

```text
scram-sha-256
```

Ответ:

> Для парольной аутентификации предпочтительно использовать `scram-sha-256`. Более слабые варианты вроде `trust` и устаревшие схемы нельзя использовать для production.

### Вопрос: нужен ли SSL для PostgreSQL?

Зависит от сети и требований безопасности.

Если база и backend в одной закрытой Docker-сети, может быть не обязательно.

Если подключение идет между серверами или через потенциально недоверенную сеть, SSL может быть нужен.

Ответ:

> Нужно уточнить требования ИБ и сетевую схему. Если трафик к базе выходит за пределы локальной машины или закрытой внутренней сети, SSL стоит рассмотреть обязательно.

### Вопрос: нужны ли бэкапы?

Для production ответ всегда да.

Ответ:

> Для production бэкапы обязательны. Нужно определить способ бэкапа, расписание, срок хранения, место хранения и процедуру восстановления.

Важная мысль:

> Бэкап считается рабочим только после тестового восстановления.

### Вопрос: что такое WAL и PITR?

WAL - write-ahead log. Это журналы изменений PostgreSQL.

PITR - point-in-time recovery. Восстановление базы на конкретный момент времени.

Простой пример:

```text
В 12:05 случайно удалили данные.
С PITR можно восстановиться на 12:04.
```

Ответ:

> Если проекту критична потеря данных, нужно обсудить WAL-архивацию и PITR. Это зона ответственности DBA/DevOps, но backend-команда должна понимать требование по восстановлению.

## 11. Что обязательно проверить в PostgreSQL-конфигах

Попроси сотрудников показать:

- `postgresql.conf`;
- `pg_hba.conf`;
- версию PostgreSQL;
- окружение, для которого сделаны настройки;
- список предположений по серверу: RAM, CPU, диск, нагрузка;
- какие IP имеют доступ;
- как применить конфиг;
- как проверить, что база работает;
- как посмотреть логи;
- как сделать бэкап;
- как восстановить бэкап хотя бы на тестовом стенде.

Минимальные команды проверки могут быть такими:

```bash
psql -h <host> -p 5432 -U <user> -d <database>
```

Проверка, что сервер доступен:

```bash
pg_isready -h <host> -p 5432
```

Перечитать конфиг без полного рестарта, если это возможно:

```sql
SELECT pg_reload_conf();
```

Посмотреть настройки:

```sql
SHOW listen_addresses;
SHOW port;
SHOW max_connections;
SHOW log_min_duration_statement;
```

## 12. Красные флаги в PostgreSQL

Стоит насторожиться, если:

- в `pg_hba.conf` есть `trust` для не локальных подключений;
- PostgreSQL открыт на `0.0.0.0/0` без строгих ограничений;
- порт `5432` доступен из интернета;
- все приложения подключаются под суперпользователем `postgres`;
- пароль прописан прямо в конфиге, README или Dockerfile;
- нет отдельных пользователей для приложения и администрирования;
- нет бэкапов для production;
- никто не проверял восстановление из бэкапа;
- параметры памяти взяты “из интернета” без учета RAM сервера;
- сильно увеличили `max_connections` без connection pool;
- нет логирования ошибок и медленных запросов;
- непонятно, где лежат данные PostgreSQL;
- непонятно, что будет при перезапуске контейнера или сервера.

## 13. Как отвечать, если ты не знаешь ответ

Не надо угадывать. Лучше фиксировать неизвестность профессионально.

Плохой ответ:

> Не знаю.

Хороший ответ:

> Сейчас этих вводных нет. Чтобы корректно подготовить конфиг, нужно уточнить: окружение, адрес backend, домен, схему сети и требования по безопасности. Я зафиксирую это как открытый вопрос.

Еще варианты:

> Это зависит от окружения. Для dev можно проще, для prod нужно согласовать с инфраструктурой и ИБ.

> Я не буду придумывать значение без данных по серверу. Для PostgreSQL нужны хотя бы RAM, CPU, диск, версия и ожидаемая нагрузка.

> Давайте разделим: что можно настроить сейчас, а что оставим параметрами до получения точных вводных.

> Для production я бы не принимал такую настройку без подтверждения ответственного за инфраструктуру.

## 14. Какие вводные нужно собрать до подготовки конфигов

Можно отправить команде такой список.

### Общие вводные

- Название сервиса.
- Окружение: dev/test/stage/prod.
- Способ запуска: ВМ, Docker Compose, Kubernetes, bare metal.
- Кто отвечает за серверы.
- Кто отвечает за безопасность.
- Где хранятся секреты.
- Как происходит деплой.
- Как выполняется откат.

### Для nginx

- Домен или домены.
- Нужен ли HTTPS.
- Источник SSL-сертификатов.
- Backend host и port.
- Нужно ли отдавать frontend-статику.
- Нужно ли проксировать `/api`.

## 17. Минимальный набор терминов

### Reverse proxy

Сервер-посредник. Принимает запросы от пользователей и передает их внутреннему приложению.

### Upstream

Backend-сервис или группа backend-сервисов, куда nginx отправляет запросы.

### HTTPS

HTTP поверх TLS. Нужен, чтобы шифровать трафик между пользователем и сервером.

### SSL/TLS-сертификат

Файл, который позволяет браузеру доверять сайту и устанавливать защищенное соединение.

### Port

Номер “двери”, на которой сервис принимает подключения. nginx часто использует `80` и `443`, PostgreSQL - `5432`.

### Host

Адрес сервера или имя сервиса.

### Firewall

Сетевой фильтр, который разрешает или запрещает подключения.

### Connection pool

Механизм переиспользования подключений к базе. Нужен, чтобы приложение не создавало слишком много подключений.

### WAL

Журнал изменений PostgreSQL. Используется для надежности, восстановления и репликации.

### Backup

Резервная копия.

### Restore

Восстановление из резервной копии.

### PITR

Восстановление базы на конкретный момент времени.

## 18. Примеры безопасных формулировок

Когда тебя спрашивают про production:

> Для production нужно отдельно подтвердить HTTPS, доступы, секреты, бэкапы, логи и процедуру отката.

Когда тебя спрашивают про параметр PostgreSQL:

> Этот параметр зависит от ресурсов сервера и нагрузки. Без RAM/CPU/диска и ожидаемого числа подключений я могу только зафиксировать стартовое предположение, но не финальное значение.

Когда тебе предлагают открыть базу наружу:

> Я бы не открывал PostgreSQL в интернет. Лучше ограничить доступ внутренней сетью, VPN или bastion host и явно прописать разрешенные IP в `pg_hba.conf`.

Когда тебе дают конфиг без пояснений:

> Давайте добавим к конфигу короткую инструкцию: назначение файла, окружение, как проверить, как применить, как откатить и какие значения нужно заменить.

Когда в конфиге есть секреты:

> Секреты не должны храниться в конфигурационном файле репозитория. Нужно вынести их в переменные окружения или секрет-хранилище.

Когда ты не уверен:

> Я уточню это у ответственного за инфраструктуру и зафиксирую как открытый вопрос, чтобы мы не приняли неверное предположение за факт.

## 20. Главное, что нужно запомнить

1. nginx принимает веб-запросы и передает их приложению.
2. PostgreSQL хранит данные приложения.
3. Для production нельзя принимать конфиги без окружения, доменов, доступов, логов, бэкапов и инструкции отката.
4. Если не знаешь ответ, не угадывай. Фиксируй открытый вопрос.
5. Безопасность важнее “лишь бы заработало”.
6. База данных не должна быть открыта наружу без очень веской причины.
7. Секреты не должны лежать в конфигурационных файлах репозитория.
8. Бэкап полезен только если восстановление проверено.
9. Хороший конфиг сопровождается инструкцией проверки, применения и отката.
10. Твоя профессиональная позиция: не знать все параметры наизусть, а уметь не пропустить рискованные места.
