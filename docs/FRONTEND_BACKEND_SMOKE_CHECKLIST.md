# Frontend + Backend Smoke Checklist (MVP)

Дата обновления: 06.03.2026

## 1. Подготовка окружения

1. Установить зависимости backend:
   - `pip install -r backend/requirements.txt`
2. Выполнить миграции и сиды:
   - `python backend/manage.py migrate`
   - `python backend/manage.py seed_references`
   - `python backend/manage.py seed_technologies`
   - `python backend/manage.py seed_users`
3. Убедиться, что в `src/js/config/api-config.local.js` включен API-режим:
   - `window.API_BASE_URL = 'http://localhost:8000';`
   - `window.USE_API = true;`
4. Запустить backend:
   - `python backend/manage.py runserver`
5. Запустить frontend локально (ваш обычный способ запуска).

## 2. Базовая доступность API

1. Открыть `GET /api/v1/health`:
   - ожидается `200`, тело с `status=ok`.
2. Открыть `GET /api/v1/openapi.json`:
   - ожидается `200`, валидный OpenAPI JSON (`openapi`, `paths`).
3. Открыть `GET /api/v1/docs`:
   - ожидается отображение Swagger UI и список endpoint.

## 3. Аутентификация и сессия

1. Логин тестовым пользователем:
   - страница `auth.html` -> ввод логина/пароля.
   - ожидается переход в 2FA flow (`requires_2fa=true`).
2. Ввод 2FA-кода:
   - ожидается успешный вход и переход в основное приложение.
3. Проверка текущего пользователя:
   - запрос `GET /api/v1/users/me/` возвращает корректные `username`, `role`, `is_2fa_enabled`.
4. Проверка logout:
   - после выхода защищенные страницы недоступны без нового логина.
5. Проверка refresh:
   - искусственно дождаться/смоделировать `401` (или вручную испортить access token) и убедиться, что frontend пытается refresh и восстанавливает сессию.

## 4. Роли и доступы

1. Пользователь `admin`:
   - имеет доступ к `/api/v1/admin-panel/*` и `/api/v1/metrics`.
2. Пользователь не `admin` (например, `analyst`):
   - получает `403` на `/api/v1/admin-panel/*` и `/api/v1/metrics`.
3. Пользователь `viewer`:
   - может читать данные, но не изменять технологии/справочники.

## 5. Технологии (CRUD + фильтры)

1. Создание технологии из UI:
   - запись появляется в списке.
2. Редактирование технологии:
   - изменения сохраняются и повторно загружаются после refresh страницы.
3. Удаление технологии:
   - запись исчезает из списка.
4. Фильтр по предприятию:
   - UI-фильтрация соответствует данным `GET /api/v1/technologies?enterpriseId=<id>`.

## 6. Справочники

1. Загрузка справочников из backend:
   - блоки, функции, направления, вендоры, интеграторы, предприятия отображаются в UI без ошибок.
2. Изменение справочника от `admin`:
   - `PUT /api/v1/references/{name}` отрабатывает успешно.
3. Попытка изменения справочника от не-`admin`:
   - ожидается `403`.

## 7. Админ-панель

1. Пользователи:
   - create/edit/delete пользователя работают.
2. Аудит:
   - действия фиксируются и отображаются в списке.
3. Бэкапы:
   - create/download/delete работают, скачивается JSON-файл.
4. Предприятия:
   - create/edit/delete работают, связи с блоками корректны.

## 8. Ошибки и UX поведения

1. Вызвать валидационную ошибку (например, пустой обязательный `name`):
   - UI показывает понятное сообщение, backend возвращает стандартизованный формат ошибки.
2. Проверить `401/403/404/429`:
   - сообщения не "сырые", UI не зависает, сценарии восстановления понятные.
3. Проверить таймаут/недоступность backend:
   - UI сообщает о проблеме сети/таймауте, не ломает состояние страницы.

## 9. Security-проверки (ручные)

1. Убедиться, что в production-конфиге:
   - `DEBUG=False`
   - задан безопасный `SECRET_KEY`
   - корректные `ALLOWED_HOSTS`
   - корректные `CORS_ALLOWED_ORIGINS`
2. Проверить rate limit на auth endpoint:
   - серия быстрых запросов к `login` или `refresh` приводит к `429`.
3. Проверить, что refresh token ротируется:
   - после успешного `refresh` старый refresh token больше не принимается.

## 10. Критерий прохождения smoke

Smoke считается пройденным, если:

1. Все пункты 2-7 проходят без блокирующих дефектов.
2. Ошибки из пункта 8 обрабатываются предсказуемо.
3. Security-пункты из раздела 9 подтверждены.
