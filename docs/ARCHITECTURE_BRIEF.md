# Документ для архитекторов: Система «Радар технологий» (РТП-3)

**Проверено:** 04.03.2026

## Назначение документа

Документ предназначен для передачи архитекторам с целью разработки архитектурной схемы системы. Вся информация получена на основе анализа кодовой базы приложения, реализованного фронтенда и проектируемого бэкенда. Документ содержит функциональные и технические требования, описание интеграций, системные требования, сведения о пользователях и перспективах развития.

## 1. Общее описание системы

### 1.1 Назначение

Система «Радар технологий» представляет собой веб-приложение для интерактивной визуализации цифровой зрелости и технологического ландшафта предприятий горно-металлургической отрасли. Основные функции:

- Интерактивная визуализация технологий на радарном графике (квадранты, кольца зрелости)
- Управление каталогом технологий: создание, редактирование, удаление
- Фильтрация по функциональным блокам, предприятиям, вендорам, направлениям
- Экспорт данных в PDF
- Администрирование пользователей, аудит действий, резервное копирование
- Ролевая модель доступа с учётом бизнес-ролей (аналитик, архитектор, директор, РП, администратор)

### 1.2 Ключевые пользовательские сценарии

1. Пользователь входит в систему через страницу авторизации, при необходимости проходит двухфакторную аутентификацию
2. Выбирает предприятие (или просматривает общий радар) и переходит к детальному просмотру
3. Изучает технологии на радаре, применяет фильтры, просматривает детали выбранной технологии
4. При наличии прав редактирует технологии, добавляет новые, удаляет устаревшие
5. Экспортирует данные в PDF для отчётности
6. Администратор управляет пользователями, просматривает журнал аудита, создаёт и восстанавливает резервные копии

## 2. Функционально-технические требования (ФТТ)

### 2.1 Аутентификация и авторизация

| Требование | Описание |
|------------|----------|
| Аутентификация по логину и паролю | Вход через POST-запрос к API с учётными данными |
| Двухфакторная аутентификация (2FA) | Поддержка TOTP (RFC 6238), настройка через QR-код, проверка 6-значного кода |
| JWT-токены | Access-токен и Refresh-токен для сессионного доступа к API |
| Обновление токенов | При 401 автоматический refresh через отдельный endpoint |
| Ролевая модель | Роли: admin, architect, director, project_manager, analyst |

### 2.2 Управление технологиями

| Требование | Описание |
|------------|----------|
| Просмотр списка технологий | GET с опциональной фильтрацией по enterpriseId |
| Создание технологии | POST с телом в формате JSON |
| Обновление технологии | PATCH или PUT по идентификатору |
| Удаление технологии | DELETE по идентификатору |
| Массовое сохранение | PUT bulk-endpoint для пакетных операций |

### 2.3 Справочники

| Справочник | Назначение |
|------------|------------|
| blocks | Функциональные блоки (например: Горное производство, Обогатительное производство) |
| functions | Функции, покрываемые технологиями |
| functionToBlock | Связь функций с блоками |
| digitalDirections | Цифровые направления (Data Hub, ИИ, автономные комплексы и т.д.) |
| directionToQuadrant | Привязка направлений к квадрантам радара |
| vendors | Список вендоров |
| integrators | Список интеграторов |
| enterprises | Предприятия (РМК, ТГОК, МГОК, КБМ, КМЭЗ, АМК) |
| enterprisesBlocksMapping | Привязка предприятий к функциональным блокам |

### 2.4 Визуализация радара

| Требование | Описание |
|------------|----------|
| SVG-рендеринг | Радар строится на Vanilla JS (DOM API), SVG viewBox 1000×1000 |
| Квадранты | Четыре квадранта по цифровым направлениям |
| Кольца зрелости | Уровни TRL/статуса: от «Исследуется» до «Внедрена» |
| Позиционирование blip | Алгоритм размещения без пересечений, spatial index (Quadtree) |
| Масштабирование | Зум по квадрантам |
| Интерактивность | Hover, click, выбор технологии, панель деталей |

### 2.5 Экспорт и отчёты

| Требование | Описание |
|------------|----------|
| Экспорт в PDF | jsPDF + html2canvas, экспорт текущего радара и данных |
| Экспорт данных | Настройка полей и фильтров для отчёта |

### 2.6 Администрирование

| Требование | Описание |
|------------|----------|
| Управление пользователями | CRUD пользователей, назначение ролей |
| Журнал аудита | Логирование входа, выхода, создания/изменения технологий, действий в админке |
| Резервные копии | Создание снимков данных (technologies, справочники, VFS), восстановление |
| Экспорт данных админки | Выгрузка данных для аналитики |
| Управление предприятиями | Редактирование enterprises и enterprises-blocks-mapping |

## 3. Функциональные и нефункциональные требования

### 3.1 Функциональные требования

- Система должна поддерживать работу в двух режимах: с backend API и в автономном mock-режиме (JSON + localStorage)
- Фильтры радара должны учитывать привязку предприятий к блокам (enterprises-blocks-mapping)
- Валидация дубликатов технологий при создании/редактировании (учёт омоглифов, регистра)
- Поддержка светлой и тёмной темы интерфейса
- Адаптивность интерфейса для мобильных устройств
- Доступность (ARIA, keyboard navigation, focus trap в модальных окнах)
- Обработка офлайн-режима с уведомлением пользователя

### 3.2 Нефункциональные требования

| Категория | Требование |
|-----------|------------|
| Производительность | Время отклика API не более 2 секунд для обычных запросов; тяжёлые операции (экспорт) — до 30 секунд |
| Таймауты | Стандартный таймаут запросов: 8 секунд; тяжёлые запросы: 30 секунд |
| Безопасность | HTTPS в production; Bearer-авторизация; CORS для разрешённых origin |
| Доступность | Поддержка скринридеров, навигация с клавиатуры |
| Совместимость | Браузеры: Chrome, Firefox, Edge, Safari (последние версии) |

### 3.3 Бизнес-требования

- Обеспечение единой «версии правды» о технологическом ландшафте предприятий
- Поддержка принятия решений директорами и руководителями проектов на основе визуализации
- Аудит всех значимых действий для соответствия корпоративным требованиям
- Возможность быстрого восстановления данных после сбоев (бэкапы)
- Масштабируемость на добавление новых предприятий и функциональных блоков без изменения архитектуры

```
┌─────────────────────┐     enterprises-blocks-mapping      ┌─────────────────────┐
│   enterprises       │◄───────────────────────────────────►│   blocks            │

│                         РАДАР (отображение технологий)                            │
```

## 5. Техническая информация о работе системы

### 5.1 Модель данных «Технология»

```
Технология:
├── id: number
├── name: string
├── description: string
├── block: number (id функционального блока)
├── blocks: number[] (при множественной привязке)
├── functionCoverage: string[] (названия функций)
├── directions: number[] (id цифровых направлений)
├── enterprises: [
│     {
│       enterpriseId: number,
│       technologicalReadiness: 1-9,
│       organizationalReadiness: 1-9,
│       status: "Внедрена" | "Невнедренна" | ...
│     }
│   ]
├── trlStage: 1-9
├── funcCover: 1-3
├── vendors: [
│     { name: string, integrators: string[] }
│   ]
├── marketExamples: string[]
└── documentationFiles: string[]
```

Внутренний (клиентский) формат после нормализации использует поля: techRead, organRead (0–3), companyRatings, level, exampleDesc (объединённые marketExamples через перевод строки).

### 5.2 Хранение в mock-режиме

| Ключ localStorage | Содержимое |
|-------------------|------------|
| vfs:technologies.json | Массив технологий |
| vfs:blocks.json | Справочник блоков |
| vfs:enterprises.json | Справочник предприятий |
| vfs:enterprises-blocks-mapping.json | Привязка предприятий к блокам |
| vfs:*.json | Остальные справочники при изменении |
| adminAuditLogs | Журнал аудита |
| isLoggedIn, username, role | Данные сессии (mock) |
| rmk_access_token, rmk_refresh_token | JWT (при API) |

### 5.3 Конфигурация API

| Параметр | Описание | Значение по умолчанию |
|----------|----------|------------------------|
| API_BASE_URL | Базовый URL backend (без завершающего /) | '' (пусто — mock) |
| USE_API | true — API, false — mock | Определяется по API_BASE_URL |
| DEFAULT_TIMEOUT_MS | Таймаут обычных запросов | 8000 |
| HEAVY_REQUEST_TIMEOUT_MS | Таймаут тяжёлых запросов | 30000 |
| TOKEN_STORAGE_KEY | Ключ access-токена | rmk_access_token |
| REFRESH_TOKEN_STORAGE_KEY | Ключ refresh-токена | rmk_refresh_token |

## 6. Системные требования (текущий стек)

### 6.1 Фронтенд

| Компонент | Технология |
|-----------|------------|
| Сборка | Vite 7.x |
| Визуализация радара | Vanilla JS (DOM API, SVG) |
| 2FA | otplib, @otplib/plugin-crypto-web, qrcode |
| PDF-экспорт | jsPDF, jspdf-autotable, html2canvas (CDN) |
| Графики в админке | Chart.js (CDN) |

## 7. Сайзинг

### 7.1 Объёмы данных (текущие/оценочные)

| Сущность | Оценка объёма |
|----------|---------------|
| Технологии | Десятки–сотни записей на холдинг |
| Функциональные блоки | 25 записей |
| Предприятия | 6+ записей |
| Цифровые направления | 4 записи |
| Вендоры, интеграторы | Десятки строк |
| Пользователи | Десятки учётных записей |
| Журнал аудита | Растёт со временем, требуется ротация/архивирование |

### 7.2 Вычислительные мощности

Текущая реализация — SPA без серверного рендеринга. Рекомендации:

- **Фронтенд**: статический хостинг (Nginx, CDN) — минимальная нагрузка
- **Backend**: ориентир на 1–2 CPU, 2–4 GB RAM для тестовой системы при нагрузке до 50–100 одновременных пользователей
- **База данных**: PostgreSQL или аналог, начальный объём данных менее 1 GB
- **Файловое хранилище**: при добавлении загрузки документов — объём зависит от политики хранения

Точный сайзинг должен быть уточнён по результатам нагрузочного тестирования и планов по количеству предприятий и технологий.

## 11. Краткое резюме для архитекторов

Система «Радар технологий» — это SPA с чётким разделением на фронтенд (Vite, Vanilla JS) и будущий REST API backend. Фронтенд поддерживает два режима: mock (JSON + localStorage) и полноценную работу с API. Ключевые интеграционные точки: аутентификация (JWT, 2FA), CRUD технологий, справочники. Данные привязаны к предприятиям и функциональным блокам. Ролевая модель определяет уровень доступа (от аналитика до администратора). Для архитектуры важно учесть: единый формат данных API↔клиент, обработку 401 с refresh токена, поддержку CORS, хранение и ротацию журнала аудита, механизм бэкапов. Система рассчитана на десятки пользователей и сотни технологий с возможностью масштабирования по данным и нагрузке.
