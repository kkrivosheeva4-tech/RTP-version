## Текущий механизм позиционирования

### Определение квадранта

**ОБНОВЛЕНО (2026-01-29):** Система использует упрощенный двухуровневый механизм определения квадранта:

1. **Основной механизм (приоритет 1)**: Направления цифрового развития
   - Маппинг: `directionToQuadrant.json`
   - Каждое направление может быть привязано к одному или нескольким квадрантам
   - Пример: "Единый центр данных (Data Hub)" → квадрант 1
   - **Это единственный способ определения квадранта для технологии**

2. **Механизм по умолчанию (приоритет 2)**: Квадрант 1
   - Если у технологии нет направлений
   - Логируется предупреждение
   - Технология все равно отображается на радаре

**Важно:** Функциональные блоки (`blocks`) больше **не используются** для определения квадранта. Они являются отдельными критериями технологии и описывают функциональные области, к которым относится технология. Блоки могут быть в любом квадранте и не влияют на позиционирование технологии на радаре.

### Расчет угла (θ)

Угол рассчитывается на основе:
- Первого направления технологии
- Золотого угла (Golden Angle = 137.50776405003785°) для равномерного распределения
- Уникального хеша на основе `tech.id` и `quadrantId`

Для технологий с несколькими направлениями (и соответственно несколькими квадрантами) каждый квадрант получает уникальный угол на основе комбинации `tech.id` и `quadrantId`.

### Расчет радиуса (r)

Радиус рассчитывается через математическую модель на основе:
- `techRead` (технологическая готовность)
- `organRead` (организационная готовность)
- `funcCover` (покрытие функций)
- `trlStage` (TRL стадия)

## Выявленные недостатки

**⚠️ ВАЖНО: Все перечисленные ниже недостатки помечены как "Реализация на будущее" и будут рассмотрены после показа приложения руководству.**

**Примечание:** Поле "Направления цифрового развития" является обязательным при добавлении и редактировании технологии, поэтому технологии без направлений не могут быть созданы через форму. Fallback механизм в коде (размещение в квадранте 1 по умолчанию) используется только для обработки существующих данных или данных, загруженных из внешних источников.

### 1. Неполная обработка технологий с несколькими направлениями ⏳ Реализация на будущее

**Проблема:**
- Технология может иметь несколько направлений
- Каждое направление может указывать на разные квадранты
- Технология отображается во всех соответствующих квадрантах
- Угол рассчитывается корректно для каждого квадранта, но может быть неочевидно для пользователя

**Код:**
```javascript
// positioning.js
const directions = Array.isArray(tech.directions) && tech.directions.length
  ? tech.directions
  : (tech.direction ? [tech.direction] : []);

directions.forEach(directionName => {
  const directionQuadrants = getQuadrantsForDirection(directionName);
  directionQuadrants.forEach(q => quadrantsSet.add(q));
});
```

**Последствия:**
- Технологии в разных квадрантах могут иметь одинаковый радиус (что правильно)
- Но разные углы (что тоже правильно)
- Может быть неочевидно, что это одна и та же технология
- Нет явной визуальной связи между копиями технологии в разных квадрантах

**Рекомендация:**
- Добавить визуальную индикацию, что технология отображается в нескольких квадрантах
- Рассмотреть возможность группировки технологий с несколькими квадрантами
- Улучшить отображение информации о технологии при наведении

### 2. Отсутствие валидации данных направлений ⏳ Реализация на будущее

**Проблема:**
- Нет проверки, существует ли направление в `directionToQuadrant.json`
- Нет проверки, валидны ли ID направлений в технологии
- Некорректные данные могут приводить к неожиданному поведению

**Пример:**
- Технология имеет `directions: [999]` (несуществующий ID)
- Система пытается найти направление по ID, не находит
- Используется квадрант 1 по умолчанию
- Нет явного сообщения об ошибке

**Последствия:**
- Технологии с невалидными направлениями могут неправильно позиционироваться
- Проблемы с данными могут оставаться незамеченными
- Усложняется отладка и поддержка

**Рекомендация:**
- Добавить валидацию направлений при загрузке данных
- Добавить предупреждения при обнаружении невалидных данных
- Рассмотреть возможность автоматического исправления или запроса у пользователя

### 3. Неоптимальное распределение углов в квадранте ⏳ Реализация на будущее

**Проблема:**
- Угол рассчитывается на основе `tech.id` и `quadrantId`
- При большом количестве технологий в квадранте может быть неравномерное распределение
- Золотой угол обеспечивает равномерность, но только при последовательном заполнении

**Код:**
```javascript
// positioning.js
const quadrantHash = id * 1000 + quadrantId * 37;
const angleOffset = (quadrantHash * GOLDEN_ANGLE) % ANGLE_SPAN;
```

**Последствия:**
- Технологии могут группироваться в определенных углах квадранта
- Недостаточное использование доступного углового пространства
- Может усложнять разведение технологий при наложениях

**Рекомендация:**
- Рассмотреть использование более сложного алгоритма распределения углов
- Улучшить использование доступного углового пространства
- Оптимизировать алгоритм разведения для лучшего распределения

### 4. Отсутствие механизма перебалансировки квадрантов ⏳ Реализация на будущее

**Проблема:**
- Нет автоматического механизма для перераспределения технологий между квадрантами
- Нет анализа загруженности квадрантов
- Нет рекомендаций по оптимальному распределению

**Последствия:**
- Неравномерная загруженность квадрантов может сохраняться
- Пользователь должен вручную корректировать данные
- Нет инструментов для анализа и оптимизации распределения

**Рекомендация:**
- Создать инструмент анализа распределения технологий по квадрантам
- Добавить визуализацию загруженности квадрантов
- Предоставить рекомендации по перебалансировке

### 5. Недостаточная обработка отсутствующих данных ⏳ Реализация на будущее

**Проблема:**
- Если у технологии отсутствуют данные о готовности (`techRead`, `organRead`, `funcCover`, `trlStage`)
- Технология позиционируется в зоне "недостаточно данных" (радиус ~85%)
- Но квадрант все равно определяется через направления

**Последствия:**
- Технологии с отсутствующими данными могут быть разбросаны по всем квадрантам
- Нет явной группировки технологий с недостаточными данными
- Может усложнять анализ и принятие решений

**Рекомендация:**
- Рассмотреть возможность группировки технологий с недостаточными данными
- Улучшить визуальную индикацию технологий с отсутствующими данными
- Добавить фильтр для технологий с недостаточными данными

```javascript
// Новая функция валидации
function validateTechQuadrantData(tech) {
  const issues = [];

  // Проверяем направления
  if (Array.isArray(tech.directions) && tech.directions.length > 0) {
    tech.directions.forEach(dir => {
```

```javascript
// Новая функция анализа
function analyzeQuadrantDistribution() {
  const technologies = window.StateAccessors?.getTechnologies() || [];
  const distribution = {
    1: { count: 0, technologies: [] },
    2: { count: 0, technologies: [] },
    3: { count: 0, technologies: [] },
    4: { count: 0, technologies: [] },
    unassigned: { count: 0, technologies: [] }
  };

  technologies.forEach(tech => {
    const quadrants = window.Positioning?.getAllQuadrantsForTech(tech) || [];
    if (quadrants.length === 0) {
      distribution.unassigned.count++;
      distribution.unassigned.technologies.push(tech.id);
    } else {
      quadrants.forEach(q => {
        if (distribution[q]) {
          distribution[q].count++;
          if (!distribution[q].technologies.includes(tech.id)) {
            distribution[q].technologies.push(tech.id);
          }
        }
      });
    }
  });

  // Вычисляем статистику
  const total = technologies.length;
  const stats = {
    total,
    distribution: {
      1: { count: distribution[1].count, percentage: (distribution[1].count / total * 100).toFixed(1) + '%' },
      2: { count: distribution[2].count, percentage: (distribution[2].count / total * 100).toFixed(1) + '%' },
      3: { count: distribution[3].count, percentage: (distribution[3].count / total * 100).toFixed(1) + '%' },
      4: { count: distribution[4].count, percentage: (distribution[4].count / total * 100).toFixed(1) + '%' },
      unassigned: { count: distribution.unassigned.count, percentage: (distribution.unassigned.count / total * 100).toFixed(1) + '%' }
    },
    balance: calculateBalance(distribution)
  };

  return { distribution, stats };
}
```

## Текущая реализация распределения углов

**ОБНОВЛЕНО (2026-01-29):** Реализован гибридный подход с группировкой по "лучам" для улучшения визуальной структуры квадрантов.

### Принцип работы

1. **Группировка по "лучам":**
   - Технологии группируются по фиксированным углам ("лучам"), образуя визуальную структуру от центра
   - Количество "лучей" определяется адаптивно: от 3 до 8, оптимально - квадратный корень от количества технологий
   - "Лучи" равномерно распределены по доступному угловому диапазону квадранта

2. **Распределение технологий:**
   - Технологии сортируются по радиусу (готовности) для группировки близких по готовности на одном "луче"
   - Распределение по "лучам" происходит циклически для равномерного заполнения
   - Технологии с разными радиусами, но на одном "луче", выстраиваются в визуальную линию от центра

3. **Небольшое смещение:**
   - Каждая технология получает небольшое детерминированное смещение (±1.5°) на основе ID и радиуса
   - Это помогает избежать точных наложений технологий на одном "луче"
   - Смещение детерминированно, поэтому позиции стабильны при перезагрузке

4. **Учет размеров технологий:**
   - Для каждой технологии вычисляется угловой размер на основе ее физического размера и радиуса
   - Максимальный угловой размер используется для определения границ квадранта
   - Границы квадранта учитывают размеры элементов для предотвращения выхода за пределы

### Функция расчета максимального количества технологий

```javascript
function calculateMaxTechnologiesInQuadrant(q, avgElementSize = 10, avgRadius = 200) {
  const POSITION_ANGLE_PAD = window.POSITION_ANGLE_PAD || 8;
  const ANGLE_SPAN = 90 - (POSITION_ANGLE_PAD * 2); // 74° доступного диапазона

  // Вычисляем угловой размер элемента на среднем радиусе
  const angularSize = calculateAngularSizeInDegrees(avgElementSize * 1.2, avgRadius);

  // Минимальный угловой зазор между элементами
  const minAngularGap = angularSize * 2;

  // Доступный диапазон с учетом размеров элементов
  const availableSpan = ANGLE_SPAN - (angularSize * 2);

  // Максимальное количество технологий
  const maxCount = Math.floor(availableSpan / minAngularGap);

  return Math.max(1, maxCount);
}
```

### Алгоритм распределения

1. **Сортировка технологий** по радиусу (от центра к краю), затем по ID
   - Обеспечивает группировку технологий с близкой готовностью на одном "луче"

2. **Вычисление углового размера** каждой технологии на ее радиусе
   - Учитывается физический размер элемента и радиус для корректных границ

3. **Определение границ квадранта** с учетом максимального углового размера
   - Границы рассчитываются с отступами и учетом размеров элементов

4. **Расчет оптимального количества "лучей"**
   - Адаптивно: минимум 3, максимум 8, оптимально √(количество технологий)

5. **Создание фиксированных углов для "лучей"**
   - "Лучи" равномерно распределены по доступному угловому диапазону

6. **Распределение технологий по "лучам"**
   - Циклическое распределение для равномерного заполнения
   - Добавление небольшого детерминированного смещения (±1.5°) на основе ID и радиуса

### Особенности

- **Визуальная структура:** Технологии выстраиваются в "лучи" от центра, создавая структурированный вид
- **Группировка по готовности:** Технологии с близкой готовностью (радиусом) группируются на одном "луче"
- **Сохранение радиуса:** Радиус технологии (определяемый готовностью) сохраняется при распределении по углу
- **Избежание наложений:** Небольшое смещение помогает избежать точных наложений технологий
