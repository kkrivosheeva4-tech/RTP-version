## Обзор системы

Система позиционирования технологий на радаре использует полярную систему координат (θ, r), где:
- **θ (угол)** - определяет квадрант и позицию внутри квадранта
- **r (радиус)** - определяет "дистанцию до внедрения" (готовность технологии)

Радиус рассчитывается на основе четырех факторов готовности:
1. **techRead** (0-3) - Технологическая готовность
2. **organRead** (0-3) - Организационная готовность
3. **funcCover** (0-3) - Покрытие функций
4. **trlStage** (1-3) - TRL стадия (Technology Readiness Level)

Угол определяется на основе направлений цифрового развития технологии.

## Математическая модель расчета радиуса

### Общая формула

Радиус технологии рассчитывается по следующей формуле:

```
r_i = r_min + (r_max - r_min) * (1 - z_i)
```

где:
- `r_i` - радиус технологии в процентах (0 < r < 100)
- `r_min` - минимальный радиус (по умолчанию 5%)
- `r_max` - максимальный радиус (по умолчанию 95%)
- `z_i` - сводный показатель готовности (0 ≤ z_i ≤ 1)

### Сводный показатель готовности (z_i)

Сводный показатель готовности вычисляется как взвешенная сумма нормализованных факторов:

```
z_i = Σ(w_k * x_ik)
```

где:
- `w_k` - вес фактора k
- `x_ik` - нормализованное значение фактора k для технологии i

### Нормализация факторов

Факторы нормализуются в диапазон [0, 1]:

1. **techRead** (0-3) → `x_techRead = techRead / 3`
2. **organRead** (0-3) → `x_organRead = organRead / 3`
3. **funcCover** (0-3) → `x_funcCover = funcCover / 3`
4. **trlStage** (1-3) → `x_trlStage = (trlStage - 1) / 2`

### Веса факторов (по умолчанию)

```javascript
weights = {
  techRead: 0.30,    // 30%
  organRead: 0.30,   // 30%
  funcCover: 0.20,   // 20%
  trlStage: 0.20     // 20%
}
```

**Важно:** Сумма весов всегда равна 1.0. Если фактор отсутствует, его вес перераспределяется между остальными факторами.

### Обработка отсутствующих данных

Если у технологии отсутствует более одного фактора, она позиционируется в зоне "недостаточно данных" с радиусом **85%**.

Если отсутствует один фактор, его вес перераспределяется между остальными факторами, и расчет продолжается.

### Пример расчета радиуса

**Дано:**
- techRead = 2.5
- organRead = 2.0
- funcCover = 2.0
- trlStage = 2
- r_min = 5%
- r_max = 95%

**Шаг 1: Нормализация факторов**
```
x_techRead = 2.5 / 3 = 0.833
x_organRead = 2.0 / 3 = 0.667
x_funcCover = 2.0 / 3 = 0.667
x_trlStage = (2 - 1) / 2 = 0.5
```

**Шаг 2: Расчет сводного показателя**
```
z_i = 0.30 * 0.833 + 0.30 * 0.667 + 0.20 * 0.667 + 0.20 * 0.5
z_i = 0.250 + 0.200 + 0.133 + 0.100
z_i = 0.683
```

**Шаг 3: Расчет радиуса**
```
r_i = 5 + (95 - 5) * (1 - 0.683)
r_i = 5 + 90 * 0.317
r_i = 5 + 28.53
r_i = 33.53%
```

**Результат:** Технология позиционируется на расстоянии 33.53% от центра радара.

### Интерпретация радиуса

- **Малый радиус (близко к центру)** - высокая готовность к внедрению
- **Большой радиус (близко к краю)** - низкая готовность к внедрению

Радиус инвертирован: чем выше готовность, тем меньше радиус (ближе к центру).

## Позиционирование технологий на радаре и в квадрантах

### Определение квадранта

Квадрант определяется **только** на основе направлений цифрового развития технологии.

**Механизм:**
1. Технология имеет массив направлений (`tech.directions`)
2. Каждое направление привязано к одному или нескольким квадрантам через маппинг `directionToQuadrant.json`
3. Технология отображается во всех квадрантах, соответствующих ее направлениям

**Важно:** Функциональные блоки (`blocks`) **не влияют** на определение квадранта. Они используются только для расчета покрытия функций.

### Расчет угла (θ)

Угол рассчитывается для каждого квадранта отдельно, обеспечивая уникальное позиционирование технологии в каждом квадранте.

**Формула:**
```javascript
quadrantHash = tech.id * 1000 + quadrantId * 37
angleOffset = (quadrantHash * GOLDEN_ANGLE) % ANGLE_SPAN
theta = startAngle + ANGLE_PAD + angleOffset
```

где:
- `GOLDEN_ANGLE = 137.50776405003785°` - золотой угол для равномерного распределения
- `ANGLE_SPAN = 90 - (ANGLE_PAD * 2)` - доступный угловой диапазон квадранта
- `ANGLE_PAD = 8°` - отступ от границ квадранта
- `startAngle` - начальный угол квадранта (0°, 90°, 180°, 270°)

**Особенности:**
- Каждый квадрант получает уникальный угол на основе комбинации `tech.id` и `quadrantId`
- Это гарантирует разные углы для одной технологии в разных квадрантах
- Углы детерминированы (стабильны при перезагрузке)

### Распределение технологий в квадранте

Технологии в квадранте распределяются равномерно по углу с учетом:
1. **Размеров элементов** - большие элементы требуют больше места
2. **Радиуса готовности** - технологии с близким радиусом группируются на одном "луче"
3. **Минимального расстояния** - технологии не должны накладываться друг на друга

**Алгоритм распределения:**
1. Технологии сортируются по радиусу (от центра к краю)
2. Определяется оптимальное количество "лучей" (от 3 до 8)
3. Технологии распределяются по "лучам" циклически
4. Каждая технология получает небольшое детерминированное смещение (±1.5°) для избежания наложений

### Обработка квадранта 4

Квадрант 4 пересекает границу 0°/360°, поэтому требует специальной обработки:
- Углы нормализуются в диапазон [270°, 360°) или [0°, 90°)
- При расчете расстояний учитывается переход через 0°

### Кеширование позиций

Позиции технологий кешируются для:
- **Производительности** - избежание повторных расчетов
- **Стабильности** - одинаковые позиции при перезагрузке страницы

Кеш сохраняется в `localStorage` и инвалидируется при изменении версии модели или данных.

## Расчет оценки "Покрытие функций" (funcCover)

### Общая концепция

Оценка `funcCover` (0-3) показывает, насколько технология покрывает необходимые функции в своих функциональных блоках.

### Входные данные

1. **functionCoverage** - массив функций, которые покрывает технология
2. **blocks** - массив ID функциональных блоков, к которым относится технология
3. **functionToBlock.json** - маппинг функций на блоки
4. **functionWeights.json** (опционально) - веса важности функций

### Алгоритм расчета

#### Шаг 1: Определение валидных функций

Фильтруются только те функции из `functionCoverage`, которые относятся к блокам технологии:

```javascript
validCoveredFunctions = functionCoverage.filter(func => {
  const funcBlocks = functionToBlock[func];
  return funcBlocks.some(blockId => tech.blocks.includes(blockId));
});
```

#### Шаг 2: Расчет взвешенной суммы всех функций в блоках

Для каждого блока технологии:
1. Определяется количество функций в блоке
2. Если используются веса важности, вычисляется взвешенная сумма функций
3. Суммируются взвешенные суммы по всем блокам

```javascript
totalWeightedFunctions = Σ(blockWeight_i)
```

где `blockWeight_i` - взвешенная сумма функций в блоке i.

#### Шаг 3: Расчет взвешенной суммы покрытых функций

```javascript
coveredWeightedSum = Σ(weight(func))
```

где `weight(func)` - вес функции (по умолчанию 1.0, если веса не используются).

#### Шаг 4: Расчет процента покрытия

```javascript
coveragePercent = coveredWeightedSum / totalWeightedFunctions
```

#### Шаг 5: Преобразование в оценку funcCover (0-3)

```javascript
funcCover = convertCoveragePercentToFuncCover(coveragePercent)
```

**Формула преобразования:**
- `coveragePercent = 0` → `funcCover = 0`
- `coveragePercent = 1.0` → `funcCover = 3`
- `0 < coveragePercent < 1.0` → используется плавное преобразование с учетом порогов

**Пороги:**
- `coveragePercent < 0.33` → `funcCover = 1` (низкое покрытие)
- `0.33 ≤ coveragePercent < 0.67` → `funcCover = 2` (среднее покрытие)
- `coveragePercent ≥ 0.67` → `funcCover = 3` (высокое покрытие)

### Пример расчета funcCover

**Дано:**
- Технология относится к блокам: [1, 2]
- Покрытые функции: ["Функция A", "Функция B", "Функция C"]
- В блоке 1: 10 функций
- В блоке 2: 8 функций
- Веса не используются

**Шаг 1: Определение валидных функций**
Предположим, что все три функции относятся к блокам 1 и 2:
```
validCoveredFunctions = ["Функция A", "Функция B", "Функция C"]
```

**Шаг 2: Расчет общей суммы функций**
```
totalWeightedFunctions = 10 + 8 = 18
```

**Шаг 3: Расчет покрытых функций**
```
coveredWeightedSum = 3
```

**Шаг 4: Расчет процента покрытия**
```
coveragePercent = 3 / 18 = 0.167 (16.7%)
```

**Шаг 5: Преобразование в оценку**
```
coveragePercent = 0.167 < 0.33
funcCover = 1
```

**Результат:** `funcCover = 1` (низкое покрытие функций)

### Учет важности функций

Если используется файл `functionWeights.json`, каждая функция имеет свой вес важности:

```javascript
// Пример functionWeights.json
{
  "Функция A": 2.0,  // Важная функция (вес 2.0)
  "Функция B": 1.0,  // Обычная функция (вес 1.0)
  "Функция C": 0.5   // Менее важная функция (вес 0.5)
}
```

При расчете покрытия учитываются веса:
- Взвешенная сумма покрытых функций = Σ(weight(func))
- Взвешенная сумма всех функций в блоках = Σ(weight(func) для всех функций в блоках)

### Fallback логика

Если блоки не указаны, используется старая логика на основе абсолютного количества функций:

```javascript
if (funcCount === 0) return 0;
if (funcCount === 1) return 1;
if (funcCount >= 2 && funcCount <= 3) return 2;
return 3; // 4+ функций
```

### Синхронный расчет

Для использования в позиционировании (где требуется синхронность) используется метод `calculateFuncCoverSync`, который работает с предзагруженными данными из `localStorage`.

### Пример 4: Технология с одним отсутствующим фактором

**Входные данные:**
- techRead = 2
- organRead = null (отсутствует)

**Расчет:**

```
availableWeights = {
  techRead: 0.30,
  organRead: 0,      // отсутствует
  funcCover: 0.20,
  trlStage: 0.20
}

totalWeight = 0.30 + 0.20 + 0.20 = 0.70
normalizationFactor = 1.0 / 0.70 = 1.429

normalizedWeights = {
  techRead: 0.30 * 1.429 = 0.429,
  funcCover: 0.20 * 1.429 = 0.286,
```

```
x_techRead = 2 / 3 = 0.667

```

3. Расчет радиуса:
```
r_i = 5 + (95 - 5) * (1 - 0.620) = 5 + 90 * 0.380 = 39.2%
```

**Результат:** Радиус = 39.2%

## Конфигурация модели

### Параметры конфигурации

Модель может быть настроена через объект `window.RadarModelConfig`:

```javascript
window.RadarModelConfig = {
  // Веса факторов
  weights: {
    techRead: 0.30,
    organRead: 0.30,
    funcCover: 0.20,
    trlStage: 0.20
  },

  // Диапазон радиуса
  r_min: 5,   // Минимальный радиус (%)
  r_max: 95,  // Максимальный радиус (%)

  // Опциональные параметры
  enableMissingDataPrediction: false  // Включить предсказание отсутствующих данных
};
```

### Изменение весов факторов

Веса можно изменить для изменения влияния факторов на позиционирование:

```javascript
// Пример: увеличение влияния технологической готовности
window.RadarModelConfig = {
  weights: {
    techRead: 0.50,    // 50% (было 30%)
    organRead: 0.20,   // 20% (было 30%)
    funcCover: 0.15,   // 15% (было 20%)
    trlStage: 0.15     // 15% (было 20%)
  }
};
```

**Важно:** Сумма весов должна быть равна 1.0. Если сумма не равна 1.0, веса автоматически нормализуются.

### Изменение диапазона радиуса

Диапазон радиуса можно изменить для более широкого или узкого распределения технологий:

```javascript
// Пример: более узкий диапазон (технологии ближе к центру)
window.RadarModelConfig = {
  r_min: 10,  // Было 5
  r_max: 80   // Было 95
};
```

## Заключение

Математическая модель позиционирования технологий обеспечивает:

1. **Объективность** - позиционирование основано на количественных показателях
2. **Гибкость** - параметры модели можно настраивать
3. **Прозрачность** - все расчеты детерминированы и воспроизводимы
4. **Стабильность** - позиции технологий сохраняются при перезагрузке

Модель учитывает:
- Технологическую и организационную готовность
- Покрытие функций в функциональных блоках
- TRL стадию технологии
- Отсутствующие данные

Позиционирование в квадрантах основано на направлениях цифрового развития и обеспечивает равномерное распределение технологий с учетом их размеров и готовности.
