## Цель плана

Цель не в том, чтобы "переписать проект красиво", а в том, чтобы поэтапно убрать реальные риски:

- расхождения между ТЗ и кодом
- слабые места доменной модели
- риски целостности данных
- неканоничные расчеты радара
- узкие места производительности
- операционные риски backup/restore и деплоя

Ниже план разбит на очереди реализации. Внутри каждого пункта есть:

- проблема
- почему это важно
- что именно делать
- ожидаемый результат

## Приоритеты

### P0. Исправить до активного развития функционала

Это блоки, которые создают риск неверных данных, расхождения API/UI, сложных регрессий и проблем в эксплуатации.

2. Нормализовать доменную модель технологий и связей.

5. Навести порядок в audit/versioning/moderation boundaries.
6. Зафиксировать безопасный контур backup/restore.

### P1. Взять сразу после стабилизации ядра

1. Довести файловый контур до production-ready состояния.
2. Реализовать backend-уведомления.

4. Закрыть тестовые пробелы.

### P2. Отложить до следующего этапа

## Этап 1. Зафиксировать архитектурный baseline

### Проблема

Часть документации уже расходится с фактическим кодом. Из-за этого команда будет спорить не о решениях, а о том, "что у нас вообще сейчас работает".

### Почему это важно

Без единого baseline следующие исправления будут делаться на ложных предположениях. Это особенно опасно для радара, модерации, backup/restore и прав доступа.

### Что делать

1. Утвердить текущую модель проекта как baseline:
   - реальные Django apps
   - текущие роли
   - текущие API `/api/v1/`
   - текущий flow 2FA
   - текущий moderation flow через `TechnologyProposal`
2. Пометить устаревшие документы как `legacy/outdated`.
3. Для ключевых контуров завести короткие RFC:
   - radar-calculation-contract
   - technology-domain-model
   - audit-vs-versioning
   - backup-restore-operating-model

### Ожидаемый результат

Команда работает от одного согласованного описания системы, а не от смеси ТЗ, старых docs и текущего кода.

## Этап 2. Канонизировать расчет радара

### Проблема

Сейчас позиция технологии на радаре считается в основном на фронтенде. Это значит:

- нет backend-источника истины
- формулы не закреплены как доменный контракт
- возможны расхождения между UI, экспортом и будущими интеграциями
- тяжелее тестировать и расследовать ошибки

### Почему это важно

Радар — это центральная бизнес-функция системы. Если его формулы "живут в JS", система визуально может быть рабочей, но архитектурно остается нестабильной.

### Что делать

1. Зафиксировать формальную модель расчета:
   - источник квадранта
   - источник кольца
   - формулу `func_cover`
   - правила для нескольких направлений
   - правила для нескольких функций
   - правила стабильности позиций
2. Перенести расчетный контур в backend как канонический сервис.
3. Оставить на фронтенде только визуализацию и layout-стабилизацию.
4. Добавить API-поля:
   - `radar_quadrants`
   - `radar_ring`
   - `radar_score`
   - при необходимости `radar_points`
5. Написать unit-тесты на формулы.
6. Написать contract-тесты: backend calculation -> frontend rendering.

### Как решать технически

Рекомендуемая схема:

- создать backend-модуль вида `technologies/services/radar.py`
- выделить DTO/serializer для радарных расчетов
- использовать единый calculator для API, экспорта и будущих аналитик
- на фронтенде оставить только чтение готовых значений и функции разведения точек

### Ожидаемый результат

Радар перестает быть "магией фронтенда" и становится тестируемой бизнес-логикой системы.

## Этап 3. Нормализовать доменную модель технологий

### Проблема

У технологии одновременно есть:

- `primary_block`
- `blocks`
- `functionCoverage`
- `directions`
- `legacy_function`

Из-за этого нет одного источника истины для части доменных связей, а некоторые данные дублируют друг друга.

### Почему это важно

Такая модель:

- усложняет валидацию
- создает неоднозначность в расчетах радара
- повышает риск несогласованных данных
- ухудшает читаемость API и БД

### Что делать

1. Определить каноническую иерархию:
   - `direction`
   - `block`
   - `function`
   - `technology`
2. Решить, какие связи остаются прямыми, а какие вычисляются.
3. Убрать или депрецировать legacy-поля, которые дублируют новые связи.
4. Формально описать, что считается обязательным:
   - должна ли технология иметь хотя бы одно направление
   - должен ли блок выводиться из функции или задаваться напрямую
   - допускается ли несколько направлений
   - допускается ли несколько блоков
5. Привести serializer-ы и API к новой модели.

### Рекомендуемое направление

Наиболее логичная целевая модель:

- `direction` остается отдельной осью для радара
- `functionCoverage` становится основным источником покрытия функций
- `block` выводится из функций там, где это возможно
- `primary_block` и `legacy_function` переводятся в режим совместимости и затем удаляются

### Ожидаемый результат

Модель технологий становится понятной, менее дублирующейся и проще для расчетов, валидации и API.

## Этап 4. Укрепить инварианты и валидацию

### Проблема

Сейчас часть ограничений живет только в Python-коде и serializer-ах. На уровне БД не хватает формальных гарантий.

### Почему это важно

Если доменные инварианты не закреплены в БД, система остается уязвимой к:

- ошибкам в коде
- частичным миграциям
- ручным исправлениям данных
- будущим интеграциям

### Что делать

1. Добавить `CheckConstraint` там, где диапазоны известны:
   - readiness 0..3
   - TRL 1..9
   - допустимые enum-значения статусов там, где они уже стабилизированы
2. Перенести критичные правила согласованности в backend validation layer.
3. Убрать свободные строковые статусы там, где уже понятен конечный список состояний.
4. Формализовать ошибки доменных нарушений.
5. Решить вопрос конкурентного редактирования:
   - добавить `version`/`updated_at`-based optimistic locking
   - возвращать контролируемую ошибку конфликта изменений

### Отдельно по статусам

Сейчас `Technology.status` слишком свободен. Нужно:

1. определить конечную state machine
2. отделить статус карточки от статуса moderation proposal
3. не смешивать публикацию сущности и согласование изменения

### Ожидаемый результат

Доменная целостность начинает гарантироваться не только "аккуратностью разработчика", но и самой системой.

## Этап 5. Развести audit, history и moderation

### Проблема

Сейчас три разные задачи частично перемешаны:

- аудит событий
- история версий данных
- редакторская модерация

`AuditLog` и `TechnologyProposal` помогают, но не заменяют полноценную историю версий.

### Почему это важно

Пока эти контуры не разведены, появляются проблемы:

- трудно расследовать изменения
- трудно объяснить пользователю "что было изменено"
- нельзя надежно восстановить конкретную предыдущую версию
- усложняется модерация и согласование diff-а

### Что делать

1. Оставить `AuditLog` как журнал событий.
2. Оставить `TechnologyProposal` как workflow модерации.
3. Добавить отдельную версионность для технологий:
   - `TechnologyVersion` или аналог
   - snapshot до/после публикации
   - diff между версиями
4. Сделать отображение "что изменилось" на основе version/diff layer, а не только payload.
5. Добавить request/correlation id в аудит и application logs.
6. Ввести маскирование чувствительных полей перед логированием.

### Ожидаемый результат

Каждый контур получает свою роль:

- аудит = кто и когда что сделал
- versioning = какие данные были в версии N
- moderation = кто предложил и кто согласовал изменение

## Этап 6. Исправить производительность тяжелых API

### Проблема

Сериализация технологий сейчас делает много дополнительных запросов. Это почти наверняка приводит к N+1 и росту времени ответа при увеличении данных.

### Почему это важно

Именно этот участок будет страдать первым при росте каталога технологий, числа предприятий и связей.

### Что делать

1. Разделить serializer-ы на:
   - list serializer
   - detail serializer
   - edit serializer
2. Вынести queryset preparation в отдельный слой:
   - `selectors.py` или `queries.py`
3. Системно применять:
   - `select_related`
   - `prefetch_related`
   - ограничение полей
4. Завести performance regression checklist для ключевых endpoint-ов.
5. На ключевые запросы сделать `EXPLAIN ANALYZE`.
6. Добавить индексы под реальные фильтры и join-сценарии.

### Где начать

С первого кандидата:

- `/api/v1/technologies/` list

Потом:

- detail технологии
- фильтрация по направлениям/предприятиям/готовности

### Ожидаемый результат

API технологий перестает деградировать при росте объема данных.

## Этап 7. Сделать файловый контур полноценным

### Проблема

Сейчас файловая часть технологий фактически не реализована как отдельная подсистема. Есть только JSON-поле со ссылками/именами.

### Почему это важно

По ТЗ файлы — отдельная функциональная область с доступами, скачиванием, удалением, аудитом и ограничениями.

### Что делать

1. Ввести отдельную модель файла технологии:
   - владелец/создатель
   - привязка к технологии
   - оригинальное имя
   - системное имя
   - MIME type
   - размер
   - checksum
   - дата загрузки
   - статус
2. Подключить storage backend.
3. Реализовать upload/download/delete API.
4. Добавить проверки:
   - размер
   - MIME/extension allowlist
   - лимиты количества
5. Решить политику удаления:
   - cascade
   - soft delete
   - cleanup orphaned files
6. Включить файлы в аудит.

### Ожидаемый результат

Файлы перестают быть "списком строк" и становятся полноценной управляемой частью системы.

## Этап 8. Реализовать backend-уведомления

### Проблема

В ТЗ уведомления предусмотрены, но в backend сейчас нет модели `Notification`, статусов, истории и доставки.

### Почему это важно

Без backend-уведомлений невозможно надежно поддержать:

- модерацию
- административные события
- системные предупреждения
- email/in-app историю

### Что делать

1. Ввести модель `Notification`.
2. Поддержать статусы:
   - unread
   - read
   - dismissed
3. Зафиксировать типы событий:
   - создано предложение
   - предложение одобрено/отклонено
   - технология обновлена
   - backup/restore завершен
   - критическая ошибка
4. Сделать delivery adapters:
   - in-app
   - email
5. Добавить user preferences на уровне уведомлений.
6. Для email лучше сразу заложить async delivery, но можно начать с простого слоя, который потом уйдет в очередь.

### Ожидаемый результат

Появляется единый notification-контур, пригодный для развития без переписывания.

## Этап 9. Укрепить backup/restore и административную безопасность

### Проблема

Backup/restore уже реализован, но это очень чувствительная зона:

- высокие привилегии
- риск случайного восстановления
- риск несовместимости схем
- риск неполного снимка

### Почему это важно

Даже рабочий backup/restore может стать источником аварии, если нет безопасной операционной модели.

### Что делать

1. Зафиксировать официальный состав backup-а:
   - какие сущности входят
   - что не входит
   - какие версии схемы поддерживаются
2. Ввести явную проверку совместимости `schema_version`.
3. Оставить `dry_run` обязательным предварительным шагом для restore в UI.
4. Добавить двухшаговое подтверждение restore.
5. Добавить журнал restore-попыток и их результатов.
6. Отделить логический backup приложения от инфраструктурного backup БД/storage.
7. Документировать границы:
   - когда пользоваться app restore
   - когда пользоваться infra backup

### Ожидаемый результат

Backup/restore становится управляемым инструментом, а не опасной кнопкой в админке.

## Этап 10. Усилить observability и логи

### Проблема

Сейчас есть базовые логи и внутренние метрики, но нет зрелой картины поведения системы в проде.

### Почему это важно

Без нормальной наблюдаемости команда поздно узнает:

- где деградирует API
- где ломается auth
- где растут ошибки
- как ведет себя backup/restore

### Что делать

1. Добавить request id / correlation id.
2. Перевести application logs в structured format.
3. Разделить:
   - application logs
   - audit logs
   - security logs
4. Добавить метрики:
   - latency по ключевым endpoint-ам
   - auth failures
   - moderation events
   - backup/restore outcomes
   - slow queries
5. Подключить error tracking наподобие Sentry.
6. Определить минимальные alert rules.

### Ожидаемый результат

Команда получает наблюдаемую систему, а не "черный ящик".

## Этап 11. Закрыть тестовые пробелы

### Проблема

Тесты есть, но они не покрывают ряд самых рискованных зон:

- канонические формулы радара
- производительность и N+1
- versioning/history
- файлы
- уведомления
- e2e ключевых сценариев

### Почему это важно

Без этого любое исправление выше будет хрупким.

### Что делать

1. Добавить unit-тесты на доменные формулы.
2. Добавить permission matrix tests.
3. Добавить contract tests на API.
4. Добавить regression tests на moderation flow.
5. Добавить performance smoke tests на список технологий.
6. Добавить e2e/smoke сценарии:
   - login + 2FA
   - создание/редактирование технологии
   - редакторское предложение + approve
   - просмотр радара
7. Для backup/restore оставить отдельный набор интеграционных тестов.

### Ожидаемый результат

Появляется минимальный safety net для следующего этапа разработки.

## Этап 12. Подтянуть CI/CD

### Проблема

Сейчас pipeline покрывает базовое качество, но не дает полной уверенности в безопасном деплое.

### Почему это важно

Чем больше логики переносится в backend и чем важнее данные, тем опаснее деплой "на честном слове".

### Что делать

1. Добавить обязательные contract/integration tests в pipeline.
2. Ввести post-deploy healthcheck.
3. Добавить smoke-check после миграций.
4. Добавить dependency/security scan.
5. Для тяжелых миграций договориться о low-downtime strategy:
   - backward-compatible migrations
   - expand/contract pattern
6. Описать ручной rollback runbook.

### Ожидаемый результат

Поставка становится предсказуемой и менее рискованной.

## Итог

Если идти самым безопасным путем, проект не нужно сейчас переписывать целиком. Правильнее сделать это как серию управляемых изменений:

1. Сначала закрепить доменную истину и расчет радара.
2. Затем убрать неоднозначности модели и слабые инварианты.
3. После этого ускорить тяжелые API и укрепить эксплуатационный контур.
4. Затем добавить отсутствующие подсистемы: файлы и уведомления.
