## 1. Текущее состояние кода

### 1.1. Технологический стек

| Элемент | Реализация |
|--------|-------------|
| Модульность | IIFE, экспорт в `window.*`, без ES modules |
| Сборка | Отсутствует (скрипты подключаются по порядку в HTML) |
| Загрузка модулей | Динамическая через `loadModule()` в `RMK-director.js`, порядок задан массивом `modules` |
| Состояние | `StateManager` + синхронизация с `window.*` для обратной совместимости |
| Зависимости | Неявные: проверки `window.X`, геттеры `getStateManager()`, `getFilters()` и т.п. |

### 1.2. Структура каталогов и объём

```
src/js/
├── config/           # конфиг модели радара (example)
├── modules/
│   ├── core/         # data-loader, state-manager, app-init, logger, dom-utils и др.
│   ├── ui/           # filters, modals, forms, detail-panel, toast, loading и др.
│   ├── radar/        # positioning, renderer, quadrants, events и др.
│   ├── business/     # auth, export, priorities
│   ├── integration/  # events (связка UI и радара)
│   ├── analytics/    # model-analytics, weight-optimizer и др.
│   └── utils/        # func-cover-utils
├── auth.js           # страница входа (mock)
├── admin.js          # админ-панель (~2100 строк)
├── RMK-director.js   # точка входа радара, загрузка модулей
├── script.js         # общий скрипт (tooltips, сайдбар, анимация радара)
├── radar-utils.js
├── audit-logger.js
└── help.js
```

**Крупные файлы (приоритет для разбиения):**

| Файл | Ориентировочный объём | Основное содержимое |
|------|------------------------|----------------------|
| `src/js/modules/core/data-loader.js` | ~1900 строк | Загрузка JSON/VFS, кэш fetch, нормализация технологий, инициализация фильтров, обработчики форм добавления/редактирования, пересчёт funcCover, initVendorsSelect |
| `src/js/modules/business/export.js` | ~1950 строк | Выбор полей экспорта, фильтры, применение к технологиям, генерация PDF (jsPDF/html2canvas), модальное окно экспорта |
| `src/js/admin.js` | ~2100 строк | Дашборд, CRUD пользователей, журнал аудита, резервные копии, предприятия, таблицы, формы, пагинация |

### 1.3. Точки входа и порядок загрузки

- **index.html** — главная (выбор предприятия, переход на радар).
- **radar.html** — подключает только `RMK-director.js`; тот асинхронно загружает все модули из списка в `loadAllModules()`.
- **auth.html** — `auth.js` (mock-авторизация).
- **admin.html** — `admin.js` (все разделы админки).
- **help.html** — `help.js`.

Порядок загрузки модулей для радара описан в `docs/MODULES_DOCUMENTATION.md`. Критичен: нарушение порядка приводит к обращению к ещё не созданным `window.*`.

## 2. Сильные стороны

- **Разделение по зонам ответственности:** core / ui / radar / business / integration / analytics — предсказуемое место для поиска логики.
- **Документация:** `MODULES_DOCUMENTATION.md` описывает порядок загрузки, StateManager, DOMCache, события — упрощает онбординг.
- **Единый источник состояния:** StateManager + StateAccessors; подписки в `state-utils.js` синхронизируют UI с состоянием.
- **Доступность (a11y):** Отдельные модули: ARIA (`aria-manager.js`), фокус-ловушка (`focus-trap.js`), клавиатурная навигация (`keyboard-nav.js`), туры (`onboarding.js`), офлайн (`offline-handler.js`).
- **Безопасность вывода:** Используется `escapeHtml` (в т.ч. из `dom-utils.js`) при вставке пользовательского контента в админке и формах.
- **Логирование и аудит:** Централизованный логгер (`logger.js`), журнал аудита (`audit-logger.js`, `appendAdminAudit`) для входа, удаления технологий и др.

## 3. Проблемы и риски (детально)

### 3.1. Архитектура и глобальное пространство

| Проблема | Где проявляется | Риск |
|----------|------------------|------|
| Десятки глобальных имён (`window.DataLoader`, `window.Filters`, `window.StateManager`, `window.Positioning` и т.д.) | Все модули в `src/js/modules/`, а также `auth.js`, `admin.js`, `script.js` | Конфликты имён, сложность тестирования и изоляции, неочевидные зависимости |
| Неявные зависимости | Вызовы вида `getStateManager()`, `getFilters()`, `getDataIndex()`, `getPositioning()` внутри data-loader и др.; проверки `typeof window.X === 'function'` | Невозможно статически проверить импорты; при изменении порядка загрузки — падения в рантайме |
| Отсутствие сборки и ESM | Нет webpack/vite/rollup; нет `import`/`export` | Нет tree-shaking, нет статического анализа зависимостей, сложнее рефакторинг и тесты |

**Файлы с наибольшим числом экспортов в `window`:**  
`integration/events.js`, `ui/form-management.js`, `ui/select-events.js`, `radar/radar-events.js`, `data-loader.js`.

### 3.2. Крупные монолитные файлы

**data-loader.js**

- Смешано: сетевая логика (fetch, кэш, VFS), бизнес-логика (нормализация технологий, маппинг предприятий), инициализация UI (фильтры, модальные селекты, tooltips для TRL/рейтингов), обработчики отправки форм добавления/редактирования, пересчёт funcCover, логика вендоров (initVendorsSelect).
- Рекомендуемое разбиение: см. этап 2 плана рефакторинга.

**export.js**

- Смешано: конфигурация полей экспорта, применение фильтров к списку технологий, построение модального окна выбора полей, генерация PDF, вспомогательные функции для мультиселектов и валидации.
- Рекомендуемое разбиение: см. этап 3 плана рефакторинга.

**admin.js**

- Смешано: дашборд, пользователи (CRUD, нормализация), аудит (пагинация, нормализация дат), бэкапы, предприятия (CRUD, загрузка из JSON), общие компоненты (таблицы, селекты, формы).
- Рекомендуемое разбиение: см. этап 4 плана рефакторинга.

### 3.3. Дублирование и рассинхрон

| Что дублируется | Где | Действие при рефакторинге |
|-----------------|-----|---------------------------|
| Заполнение фильтров и модальных селектов | `data-loader.js`: `loadData()` (блок initFiltersWithRetry, tooltips для TRL/рейтингов), `initFilters()` в том же файле; частично в `script.js` | Вынести единую функцию/модуль инициализации фильтров и селектов; вызывать из одного места |
| Список пользователей и ролей | `auth.js` — массив `users` для входа; `admin.js` — `getDefaultUsers()` для отображения в админке | Единый источник (конфиг или позже API); auth и admin только читают |
| Названия полей и опции (TRL, рейтинги 0–3) | data-loader: константы и tooltips для форм добавления и редактирования повторяются | Вынести в общий конфиг/константы (например `form-options.js` или в конфиг полей) |
| Очистка/нормализация данных технологий | Нормализация в data-loader при загрузке; частично в формах при submit | Сохранить одну точку нормализации (например слой адаптера данных) |

### 3.4. Состояние и обратная совместимость

- В `data-loader.js` при `setState()` дублируется запись в `window`: например `window.technologies`, `window.enterpriseData`, `window.blocksList`, `window.functions` и др. Часть кода (в т.ч. в radar, ui) читает из `window`, часть из StateManager. При переходе на API нужно либо поддерживать оба пути, либо постепенно перевести всё на state и убрать дубли в `window`.
- Опечатка в имени файла: используется `bloks.json` (и fallback на `blocks.json`). В коде и комментариях встречаются оба варианта. Рекомендация: зафиксировать один (например `blocks.json`) и заменить все упоминания `bloks` на `blocks` (включая ключи VFS при необходимости).

### 3.5. Обработка ошибок

- Нет единого контракта: в одних местах только `Logger.warn`, в других — Toast или ErrorDisplay, в третьих — пустой catch. Пользователь не всегда видит сообщение об ошибке; разработчик не всегда логирует.
- Асинхронные сценарии: отмена при повторном вызове `loadData()` или при уходе со страницы не обрабатывается; AbortController используется только внутри fetch в data-loader. Рекомендация: ввести единый слой (например `handleAsyncError(err, context)`) и по возможности прокидывать AbortSignal в длинные цепочки.

### 3.6. Тестирование

- В репозитории не обнаружены юнит- или интеграционные тесты для фронтенда. Крупные файлы и зависимость от `window` затрудняют добавление тестов без предварительного выделения чистых функций и внедрения зависимостей.

## 4. План рефакторинга по этапам

Цель: улучшить поддерживаемость и подготовить код к подключению API без «большого взрыва». Каждый этап завершается проверкой: приложение запускается, ключевые сценарии (загрузка радара, фильтры, добавление/редактирование технологии, экспорт PDF, админка) работают.

### Этап 0. Подготовка (до начала рефакторинга)

- [x] Зафиксировать текущее поведение: перечень сценариев (загрузка, фильтры, CRUD технологии, экспорт, вход, админка) и при необходимости записать короткие шаги для ручной проверки.
- [x] Убедиться, что документация модулей актуальна (`MODULES_DOCUMENTATION.md`).
- [x] Решить вопрос с именем: везде перейти на `blocks.json` (и один ключ VFS) или официально зафиксировать `bloks.json` и привести комментарии/пути к одному виду. Внести правки в один коммит.

**Результат:** Есть чек-лист для регрессии (`docs/REGRESSION_CHECKLIST.md`) и единое имя для блоков (`blocks.json`, один ключ VFS).

### Этап 1. Единый контракт ошибок и утилита показа ошибок ✅ Выполнен

**Цель:** Один способ показа ошибки пользователю и один способ логирования, чтобы при добавлении API не размазывать обработку по всем файлам.

**Шаги:**

1. Ввести в `core/` (или в `ui/`) модуль, например `error-handler.js` (или расширить существующую утилиту), который экспортирует:
   - `reportError(error, context)` — логирует через `Logger` (если есть) и при необходимости показывает пользователю через Toast/ErrorDisplay; `context` — строка или объект (например `{ action: 'loadData' }`).
   - Опционально: `reportValidationErrors(fieldErrors)` для ошибок валидации форм.
2. Заменить разрозненные места с пустым catch или только `Logger.warn` на вызов `reportError` там, где ошибка касается пользовательского сценария (загрузка данных, отправка формы, экспорт). Внутренние/ожидаемые ошибки (например «файл не найден в VFS») можно по-прежнему только логировать.
3. Документировать в `MODULES_DOCUMENTATION.md`: «Показ ошибок пользователю и логирование — через reportError».

**Файлы для правок:**  
`data-loader.js` (catch в loadData, в формах), `export.js`, `admin.js`, при необходимости `app-init.js`, `audit-logger.js`.

**Критерий приёмки:** Нет «глухих» catch для сценариев, важных для пользователя; везде используется одна точка входа для отображения и логирования ошибок.

### Этап 2. Разбиение data-loader.js ✅ Выполнен

**Цель:** Разделить загрузку данных, нормализацию и инициализацию UI (фильтры, формы), чтобы упростить последующее подключение API.

**Шаги:**

1. **Выделить слой работы с данными (сетевой + VFS):**
   - Создать файл, например `core/data-source.js` (или `core/data-fetch.js`).
   - Перенести: `vfsKey`, `vfsRead`, `vfsWrite`, `clearVfsCache`, `fetchJsonWithCache`, `clearFetchCache`, `loadJsonPreferVfs`, логику `tryFetchAndParse`. Всё, что касается «получить JSON откуда-то» (сеть или VFS), — в этом модуле.
   - Экспортировать в `window.DataSource` (или сохранить текущие имена `vfsRead`, `vfsWrite`, `fetchJsonWithCache`, `loadJsonPreferVfs` в window для обратной совместимости). Вызывать из data-loader только методы этого модуля.

2. **Выделить нормализацию технологий и справочников:**
   - Создать файл, например `core/data-normalize.js`.
   - Перенести: `normalizeTechnologyFromNewFormat`, формирование `blockIdToName`, `nameToBlockId`, `enterpriseData` из массива технологий, а также константы/хелперы для RINGS, QUADRANTS, levelToRing (если они относятся только к нормализации). data-loader после получения «сырых» данных вызывает нормализацию из этого модуля и записывает результат в state.

3. **Выделить инициализацию фильтров и модальных селектов:**
   - Создать файл, например `ui/filter-init.js` (или расширить `filters.js`).
   - Перенести: всю логику `initFiltersWithRetry`, заполнение sidebar-фильтров и модальных селектов, добавление tooltips для TRL и рейтингов в одну-две функции (например `initSidebarFilters(state)`, `initModalSelects(state)`). Константы (trlOptions, ratingOptions, tooltips для полей) вынести в общий объект/файл (например `form-field-options.js`).
   - В data-loader в `loadData()` после загрузки и записи в state вызывать только эти функции инициализации фильтров, не держать там сотни строк разметки опций.

4. **Оставить в data-loader только оркестрацию:**
   - `loadData()`: вызов DataSource/loadJsonPreferVfs → вызов нормализации → setState → вызов инициализации фильтров (из filter-init) → пересчёт funcCover и координат, nextId, обработчики форм.
   - Обработчики отправки форм добавления/редактирования можно оставить в data-loader на этом этапе или вынести в `form-management.js`/отдельный модуль (см. этап 5 при желании).
   - Функции `ensureAndPersistNewTech`, `switchEnterprise`, `initFilters`, `recalculateFuncCoverForAllTechnologies` остаются в data-loader или частично переносятся в data-normalize / DataSource по смыслу.
   - `initVendorsSelect` можно перенести в `ui/vendors-files.js` или оставить в data-loader до этапа API.

5. Обновить порядок загрузки в `RMK-director.js`: подключить `data-source.js` и `data-normalize.js` до data-loader; `filter-init.js` — после filters.js. Обновить `MODULES_DOCUMENTATION.md`.

**Критерий приёмки:** data-loader сокращён до разумного объёма (ориентир — не более 600–800 строк оркестрации); загрузка, нормализация и инициализация фильтров вынесены в отдельные модули; регрессии по сценариям нет.
