## 1. Изменение деления радара: с функциональных блоков на направления цифрового развития

### 1.1. Анализ требований
- **Файл-источник:** `draft vectors digital.pdf` (найден в `docs/`, изучен)
- **Текущее состояние:**
  - Радар делится по функциональным блокам (25 блоков из `bloks.json`)
  - Функциональные блоки определяют квадрант (угол θ) через `blockToQuadrant.json`
  - Позиция технологии на радаре зависит от блока
  - 4 квадранта: "Корпоративное управление и администрация", "Основное производство", "Производственная поддержка и безопасность", "Внешние бизнесы"
- **Целевое состояние:**
  - Радар делится по **4 направлениям цифрового развития** (направления определяют квадранты)
  - **Функциональные блоки НЕ связаны с секторами/квадрантами**
  - Функциональные блоки используются **только как критерий для фильтрации и классификации технологий**
  - Позиция технологии на радаре (угол θ) определяется **направлением**, а не блоком
  - Радиус (r) остается без изменений (зависит от оценок готовности)

### 1.1.1. Направления цифрового развития (новые квадранты)

Из файла `draft vectors digital.pdf` определены **4 основных направления**, которые станут квадрантами радара:

1. **Единый центр данных (Data Hub)**
   - Описание: Единая точка управления всей компанией в реальном времени и «одна версия правды» для всех процессов и подразделений
   - Квадрант: 1 (0°–90°)

2. **Искусственный интеллект во всех процессах**
   - Описание: Широкое внедрение ИИ-решений на всех уровнях деятельности компании
   - Квадрант: 2 (90°–180°)

3. **Автономные комплексы на всех предприятиях**
   - Описание: Создание «light out» (безлюдных или минимально обслуживаемых) участков и предприятий, функционирующих в автономном режиме
   - Квадрант: 3 (180°–270°)

4. **Надёжная и безопасная цифровая инфраструктура**
   - Описание: Необходима для масштабирования решений на основе искусственного интеллекта и автономных систем, обеспечивая устойчивость, безопасность и производительность
   - Квадрант: 4 (270°–360°)

### 1.2. Ключевые изменения архитектуры

**Важно:** Функциональные блоки и направления цифрового развития — это **независимые сущности**:
- **Направления** → определяют квадранты радара (угол θ) → определяют позицию технологии
- **Функциональные блоки** → используются только для фильтрации и классификации → НЕ влияют на позицию

## 3. Фильтрация блоков и функций по выбранному предприятию

### 3.1. Требование
При выборе предприятия в фильтре, в фильтрах "Функциональный блок" и "Функция" должны отображаться только те пункты, которые относятся к выбранному предприятию.

**Важно:** Функциональные блоки используются только для фильтрации, они **не связаны** с квадрантами/направлениями радара.

### 3.1.1. Связь предприятий и функциональных блоков
- **Для каждого предприятия могут быть свои функциональные блоки:**
  - Разные предприятия могут иметь разные наборы функциональных блоков
  - Один и тот же блок может присутствовать в нескольких предприятиях
  - Некоторые функциональные блоки могут **не быть ни в одном предприятии** (общие блоки или блоки без привязки)
- **Связь определяется через технологии:**
  - Если технология принадлежит предприятию и имеет функциональный блок → блок относится к этому предприятию
  - Если блок не используется ни в одной технологии предприятия → блок не показывается в фильтре для этого предприятия
  - Если блок используется в технологиях, но не привязан к конкретным предприятиям → блок может быть общим (показываться для всех или ни для кого)

## Заметки и вопросы

1. **Файл `draft vectors digital.pdf`:** Найден в `docs/`, требуется изучить для понимания структуры направлений цифрового развития.

2. **КРИТИЧЕСКОЕ ИЗМЕНЕНИЕ АРХИТЕКТУРЫ:**
   - **Направления цифрового развития** определяют квадранты радара (угол θ) → определяют позицию технологии
   - **Функциональные блоки** НЕ связаны с секторами/квадрантами
   - **Функциональные блоки** используются ТОЛЬКО для фильтрации и классификации технологий
   - При реализации необходимо полностью убрать логику определения квадранта по блоку из `positioning.js`
   - Поле `blocks` в данных технологий сохраняется, но НЕ используется для позиционирования

2. **Структура данных предприятий:** Требуется уточнить, как хранить данные о технологиях с несколькими предприятиями:
   - Вариант 1: Одна запись технологии с массивом предприятий
   - Вариант 2: Отдельные записи для каждого предприятия с общим идентификатором

3. **Обратная совместимость:** Требуется решить, нужно ли поддерживать старую структуру данных или можно сделать полную миграцию.

4. **Размеры файлов:** Требуется уточнить конкретные лимиты для файлов (максимальный размер одного файла, общий размер).

5. **Автоматический расчет покрытия функций:** Требуется уточнить точную формулу расчета на основе количества выбранных функций.
