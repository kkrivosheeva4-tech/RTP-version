## P0 (критично): безопасность и доверие к данным

### 1) Пароли и роли "зашиты" на фронтенде

**Где:** `src/js/auth.js` (немодульный), массив `users = [{username, password, role}, ...]`
**Проблема:**

- Любой пользователь может открыть DevTools и увидеть пароли/логику.
- Роль хранится в `localStorage` (`role`, `isLoggedIn`, `username`) и используется для доступа к функционалу (включая админку) → можно вручную поставить `role=admin`.
  **Риски:** полный обход авторизации, подмена данных, компрометация доверия к отчётам/аудиту.
  **Рекомендации:**
- **Минимум (для демо/офлайн):** явно пометить режим "DEMO", убрать "пароли" как секрет (например, один PIN на окружение) и добавить предупреждение в UI, что это не security boundary.
- **Правильно (prod):** вынести auth/roles на backend (JWT/session cookie), на фронте хранить только токен/сессию, роли приходят с сервера; операции записи/изменения валидируются на сервере.

### 7) `EventManager.off()` не снимает реальные слушатели

**Статус: ✅ ИСПРАВЛЕНО**

**Где:** `src/js/modules/core/core-utils.js`

**Проблема:** `EventManager.on()` создавал `document.addEventListener(...)` один раз на пару (selector,event), но `off()` только чистил Set обработчиков, не снимая базовый listener.

**Риск:** утечки/накопление "пустых" обработчиков при реинициализациях (особенно если будет SPA-навигация).

**Выполнено:**

- ✅ Добавлено хранилище `documentListeners` для отслеживания реальных слушателей.
- ✅ `EventManager.off()` теперь корректно снимает слушатели с `document` через `removeEventListener`, когда Set обработчиков становится пустым.
- ✅ `EventManager.clear()` теперь снимает все слушатели с `document` перед очисткой.

**Результат:** Устранены утечки памяти, слушатели корректно удаляются при вызове `off()` и `clear()`.
