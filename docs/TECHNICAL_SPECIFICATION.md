## 2. ЦЕЛИ И НАЗНАЧЕНИЕ ПРОЕКТА

### 2.1. Цель проекта

Разработка веб-сервиса для интерактивной визуализации, анализа и управления технологическим ландшафтом холдинга РМК с целью:

- Создания единой централизованной системы учета и визуализации технологий
- Обеспечения объективной оценки готовности технологий к внедрению на основе математической модели
- Повышения эффективности принятия стратегических решений в области цифровой трансформации
- Обеспечения прозрачности технологического ландшафта для всех уровней управления
- Автоматизации процессов сбора, анализа и представления данных о технологиях

### 2.2. Назначение проекта

Веб-сервис «Радар технологий» предназначен для:

1. **Визуализации технологического ландшафта:**
   - Интерактивное представление технологий в виде радара с квадрантами и кольцами готовности
   - Отображение позиции технологий на основе объективных критериев готовности
   - Многомерный анализ технологий по различным параметрам

2. **Управления технологиями:**
   - Добавление, редактирование и удаление технологий
   - Управление функциональными блоками и функциями
   - Привязка технологий к предприятиям холдинга
   - Управление вендорами и интеграторами

3. **Анализа и отчетности:**
   - Фильтрация и поиск технологий по различным критериям
   - Экспорт данных в различных форматах (PDF, CSV, JSON)
   - Формирование аналитических отчетов
   - Сравнительный анализ технологий по предприятиям

4. **Административного управления:**
   - Управление пользователями и ролями
   - Журналирование всех действий в системе (аудит)
   - Управление резервными копиями данных
   - Мониторинг активности пользователей

5. **Обеспечения безопасности:**
   - Разграничение прав доступа по ролям
   - Аутентификация и авторизация пользователей
   - Двухфакторная аутентификация (2FA)
   - Защита от несанкционированного доступа

## 3. ХАРАКТЕРИСТИКА ОБЪЕКТОВ АВТОМАТИЗАЦИИ

### 3.1. Объекты автоматизации

Веб-сервис автоматизирует следующие процессы:

1. **Учет и каталогизация технологий:**
   - Регистрация технологий с полным описанием характеристик
   - Классификация технологий по направлениям цифрового развития
   - Привязка технологий к функциональным блокам и функциям
   - Связывание технологий с предприятиями холдинга

2. **Оценка готовности технологий:**
   - Расчет технологической готовности (0-3)
   - Расчет организационной готовности (0-3)
   - Расчет покрытия функций (0-3)
   - Определение TRL-стадии (Technology Readiness Level)
   - Автоматический расчет позиции технологии на радаре

3. **Визуализация данных:**
   - Отображение технологий на интерактивном радаре
   - Фильтрация и поиск технологий
   - Детальное представление информации о технологиях
   - Графическая аналитика и статистика

4. **Управление пользователями:**
   - Регистрация и аутентификация пользователей
   - Управление ролями и правами доступа
   - Мониторинг активности пользователей
   - Журналирование действий

5. **Формирование отчетности:**
   - Генерация отчетов по технологиям
   - Экспорт данных в различных форматах
   - Аналитические сводки и статистика

### 3.2. Пользователи системы

Система предназначена для следующих категорий пользователей:

1. **Гости (guest):**
   - Просмотр радара технологий
   - Просмотр детальной информации о технологиях
   - Использование фильтров и поиска
   - Ограниченный доступ к функциям редактирования

2. **Архитекторы (architect):**
   - Все права гостя
   - Добавление, редактирование и удаление технологий
   - Управление функциональными блоками
   - Доступ к административной панели
   - Просмотр журнала аудита
   - Управление пользователями
   - Экспорт данных в PDF

3. **Директора (director):**
   - Все права архитектора
   - Доступ к расширенной странице радара
   - Управление вендорами и интеграторами
   - Прикрепление файлов к технологиям
   - Расширенная аналитика

4. **Руководители проектов (project_manager):**
   - Аналогичны правам директора
   - Управление приоритетами технологий
   - Расширенная аналитика проектов

5. **Администраторы (admin):**
   - Полный доступ ко всем функциям системы
   - Управление резервными копиями
   - Экспорт всех данных (пользователи, аудит, технологии)
   - Управление предприятиями
   - Полный доступ к административной панели

## 4. ТРЕБОВАНИЯ К ВЕБ-СЕРВИСУ

### 4.1. Функциональные требования

#### 4.1.1. Модуль визуализации радара

**ФР-1.1. Отображение радара технологий:**
- Система должна отображать интерактивный радар с технологиями
- Радар должен быть разделен на 4 квадранта, соответствующих направлениям цифрового развития:
  - Квадрант 1: Единый центр данных (Data Hub)
  - Квадрант 2: Искусственный интеллект во всех процессах
  - Квадрант 3: Автономные комплексы на всех предприятиях
  - Квадрант 4: Надёжная и безопасная цифровая инфраструктура
- Технологии должны отображаться в виде кругов, размер которых зависит от количества вендоров
- Позиция технологии на радаре должна определяться математической моделью на основе факторов готовности

**ФР-1.2. Интерактивность радара:**
- При клике на квадрант должен открываться зум (увеличение масштаба)
- При зуме должна открываться правая панель со списком технологий в квадранте
- При клике на технологию должна открываться детальная панель с полной информацией
- Должна быть возможность навигации по радару (масштабирование, перемещение)

**ФР-1.3. Математическая модель позиционирования:**
- Система должна рассчитывать позицию технологии на радаре на основе четырех факторов:
  - Технологическая готовность (techRead): 0-3
  - Организационная готовность (organRead): 0-3
  - Покрытие функций (funcCover): 0-3 (автоматический расчет)
  - TRL-стадия (trlStage): 1-3
- Радиус технологии должен рассчитываться по формуле: `r = r_min + (r_max - r_min) * (1 - z)`, где z - сводный показатель готовности
- Угол технологии должен определяться на основе направлений цифрового развития
- Чем выше готовность технологии, тем ближе к центру она должна располагаться

#### 4.1.2. Модуль управления технологиями

**ФР-2.1. Добавление технологии:**
- Система должна предоставлять форму для добавления новой технологии
- Обязательные поля:
  - Название технологии
  - Направления цифрового развития (можно выбрать несколько)
  - Функциональный блок
  - Функции (можно выбрать несколько)
  - TRL-стадия (1-Исследовательская, 2-Прототип, 3-Готова к внедрению)
  - Описание
  - Выбор предприятий (или галочка "Технология применима в холдинге")
- Необязательные поля:
  - Примеры внедрений на рынке или в контуре
  - Вендоры и интеграторы
  - Прикрепленные файлы
- Для каждого предприятия должны быть указаны оценки:
  - Технологическая готовность (0-3)
  - Организационная готовность (0-3)
- Система должна автоматически рассчитывать "Покрытие функций" на основе выбранных функций и блоков

**ФР-2.2. Редактирование технологии:**
- Система должна предоставлять форму редактирования с предзаполненными данными
- Все поля должны быть доступны для изменения (с учетом прав доступа)
- При изменении оценок готовности должна автоматически пересчитываться позиция технологии на радаре

**ФР-2.3. Удаление технологии:**
- Система должна предоставлять возможность удаления технологии
- Удаление должно требовать подтверждения
- При удалении должна создаваться запись в журнале аудита

**ФР-2.4. Просмотр детальной информации:**
- При клике на технологию должна открываться детальная панель с информацией:
  - Название технологии
  - Функциональный блок и функции
  - TRL-стадия
  - Покрытие функций (визуальное отображение)
  - Оценки по предприятиям (технологическая и организационная готовность)
  - Описание
  - Вендоры и интеграторы
  - Примеры внедрений
  - Прикрепленные файлы

#### 4.1.3. Модуль фильтрации и поиска

**ФР-3.1. Поиск технологий:**
- Система должна предоставлять поле поиска для поиска технологий по названию
- Поиск должен работать в реальном времени
- Найденные технологии должны подсвечиваться на радаре
- Технологии, не соответствующие запросу, должны скрываться

**ФР-3.2. Фильтрация технологий:**
- Система должна предоставлять фильтры:
  - По предприятиям (множественный выбор)
  - По функциональным блокам (множественный выбор)
  - По функциям (множественный выбор)
  - По статусу внедрения (Внедренные/Невнедренные)
- Фильтры должны работать совместно (логика "И")
- Должна быть возможность сброса всех фильтров

**ФР-3.3. Комбинирование поиска и фильтров:**
- Поиск и фильтры должны работать совместно
- Технология должна отображаться только если соответствует и поисковому запросу, и всем примененным фильтрам

#### 4.1.4. Модуль управления функциональными блоками

**ФР-4.1. Добавление функционального блока:**
- Система должна предоставлять форму для добавления функционального блока
- Обязательные поля:
  - Название блока
  - Направление цифрового развития
  - Функции (минимум 1, максимум 20)
- После добавления блок должен стать доступен в фильтрах и формах технологий

**ФР-4.2. Редактирование и удаление функционального блока:**
- Система должна предоставлять возможность редактирования и удаления блоков
- При удалении блока должны проверяться связи с технологиями

#### 4.1.5. Модуль авторизации и аутентификации

**ФР-5.1. Вход в систему:**
- Система должна предоставлять страницу авторизации
- Поля формы:
  - Имя пользователя (обязательное)
  - Пароль (обязательное, минимум 4 символа)
  - Чекбокс "Запомнить меня"
  - Ссылка "Забыли пароль?"
- Должна быть возможность входа как гость (без авторизации)

**ФР-5.2. Двухфакторная аутентификация (2FA):**
- Система должна поддерживать двухфакторную аутентификацию через TOTP (Time-based One-Time Password)
- При первом входе пользователь должен настроить 2FA через QR-код
- При входе должен запрашиваться код из приложения-аутентификатора
- Должна быть возможность отключения 2FA администратором

**ФР-5.3. Управление сессией:**
- Система должна сохранять сессию пользователя при выборе "Запомнить меня"
- Должен быть автоматический выход при истечении времени сессии
- При выходе должны очищаться данные сессии

**ФР-5.4. Разграничение прав доступа:**
- Система должна проверять права доступа на каждой странице
- Доступ к функциям должен определяться ролью пользователя
- При отсутствии прав должен отображаться соответствующий интерфейс

#### 4.1.6. Модуль административной панели

**ФР-6.1. Управление пользователями:**
- Система должна предоставлять интерфейс для управления пользователями
- Функции:
  - Просмотр списка пользователей
  - Поиск пользователей по имени или email
  - Фильтрация по роли
  - Добавление нового пользователя
  - Редактирование пользователя (изменение роли)
  - Удаление пользователя (с подтверждением)
- Таблица должна отображать: ID, Имя, Email, Роль, Дата регистрации, Статус

**ФР-6.2. Журнал аудита:**
- Система должна вести журнал всех действий пользователей
- Записи должны содержать:
  - Дата и время
  - Пользователь
  - Тип действия (login, logout, create, update, delete, export)
  - Детали действия
  - IP адрес
- Должна быть возможность фильтрации:
  - По диапазону дат
  - По пользователю
  - По типу действия
- Должна быть пагинация для больших объемов данных
- Должна быть возможность очистки журнала (только для администраторов)

**ФР-6.3. Резервное копирование:**
- Система должна предоставлять возможность создания резервных копий
- Резервная копия должна включать:
  - Пользователи
  - Технологии
  - Функциональные блоки
  - Журнал аудита
- Должна быть возможность восстановления из резервной копии
- Должна быть возможность удаления резервных копий
- Список копий должен отображать: Название, Дата создания, Размер

**ФР-6.4. Экспорт данных:**
- Система должна предоставлять возможность экспорта:
  - Пользователей (CSV, JSON)
  - Журнала аудита (CSV, JSON)
- Экспорт должен содержать все соответствующие поля

**ФР-6.5. Управление предприятиями:**
- Система должна предоставлять интерфейс для управления предприятиями
- Функции:
  - Просмотр списка предприятий
  - Поиск предприятий
  - Добавление предприятия
  - Редактирование предприятия
  - Удаление предприятия (с проверкой связей)

**ФР-6.6. Обзорная панель (Dashboard):**
- Система должна отображать сводную статистику:
  - Общее количество пользователей
  - Количество активных сессий
  - Количество событий аудита
  - Количество резервных копий
- Должны быть графики:
  - График регистраций за последние 7 дней
  - График активности по типам действий
  - Распределение пользователей по ролям

#### 4.1.7. Модуль экспорта и отчетности

**ФР-7.1. Создание отчетов:**
- Система должна предоставлять модальное окно для выбора полей экспорта
- Доступные поля:
  - Название технологии
  - Предприятия
  - Функциональный блок
  - Функции
  - Статус
  - Технологическая готовность
  - Организационная готовность
  - TRL-стадия
  - Вендоры
  - Интеграторы
  - Описание
  - Примеры внедрения
- Для каждого поля должны быть доступны фильтры (где применимо)
- Должна быть возможность выбора всех полей одновременно

**ФР-7.2. Экспорт в PDF:**
- Система должна генерировать PDF отчет с выбранными полями
- Отчет должен содержать:
  - Заголовок "Отчет по технологиям"
  - Дата создания
  - Таблица данных
  - Информация о примененных фильтрах
  - Статистика (количество технологий)
- Файл должен автоматически скачиваться с именем `radar_report_YYYY-MM-DD_HH-mm-ss.pdf`

#### 4.1.8. Модуль уведомлений

**ФР-8.1. Система уведомлений:**
- Система должна отображать уведомления о важных событиях
- Типы уведомлений:
  - Успех (зеленый) - успешное выполнение действия
  - Информация (синий) - информационные сообщения
  - Предупреждение (желтый) - предупреждения

## 5. СИСТЕМНАЯ АРХИТЕКТУРА

### 5.1. Обоснование выбора технологического стека

#### 5.1.1. Frontend: HTML5, CSS3, JavaScript ES6+

**Обоснование:**
- **Универсальность:** Работает во всех современных браузерах без дополнительных зависимостей
- **Производительность:** Отсутствие накладных расходов фреймворков
- **Простота развертывания:** Не требует сборки (build process)
- **Легкость поддержки:** Понятный код для команды разработки
- **Модульность:** ES6 модули обеспечивают хорошую организацию кода

**Альтернативы, рассмотренные и отклоненные:**
- React/Vue/Angular: Избыточны для данного проекта, увеличивают сложность
- TypeScript: Не требуется для проекта данного масштаба

#### 5.1.2. Backend: Django + Django REST Framework

**Обоснование:**
- **Быстрая разработка:** Django предоставляет множество готовых компонентов
- **Безопасность:** Встроенная защита от распространенных уязвимостей
- **ORM:** Удобная работа с базой данных
- **Админ-панель:** Готовая административная панель для управления данными
- **REST API:** DRF обеспечивает быструю разработку API
- **Сообщество:** Большое сообщество и обширная документация

**Альтернативы, рассмотренные и отклоненные:**
- Flask: Требует больше кода для реализации функциональности
- FastAPI: Меньше готовых компонентов, больше ручной работы

#### 5.1.3. База данных: PostgreSQL

**Обоснование:**
- **Надежность:** Проверенная временем СУБД
- **Производительность:** Высокая производительность для сложных запросов
- **Функциональность:** Богатый набор функций (JSON, полнотекстовый поиск)
- **Масштабируемость:** Хорошо масштабируется
- **Open Source:** Бесплатная лицензия

**Альтернативы, рассмотренные и отклоненные:**
- MySQL: Меньше функций, проблемы с производительностью на больших объемах
- SQLite: Не подходит для продакшена с множественными пользователями

#### 5.1.4. Аутентификация: JWT (JSON Web Tokens)

**Обоснование:**
- **Stateless:** Не требует хранения сессий на сервере
- **Масштабируемость:** Легко масштабируется
- **Безопасность:** Современный стандарт аутентификации
- **Мобильность:** Легко использовать в мобильных приложениях (при необходимости)

**Альтернативы, рассмотренные и отклоненные:**
- Session-based: Требует хранения сессий, сложнее масштабировать
- OAuth2: Избыточен для внутренней системы

### 5.4. API и интеграции

#### 5.4.1. REST API Endpoints

- `POST /api/v1/auth/login/` - Вход в систему
- `POST /api/v1/auth/logout/` - Выход из системы
- `POST /api/v1/auth/refresh/` - Обновление JWT токена
- `POST /api/v1/auth/2fa/setup/` - Настройка 2FA
- `POST /api/v1/auth/2fa/verify/` - Проверка 2FA кода

- `GET /api/v1/technologies/` - Список технологий (с фильтрацией и пагинацией)
- `GET /api/v1/technologies/{id}/` - Детали технологии

### Приложение В. Математическая модель позиционирования технологий

#### В.1. Общая концепция

Математическая модель использует полярную систему координат (θ, r):
- **θ (угол)** - определяет квадрант и позицию внутри квадранта
- **r (радиус)** - определяет "дистанцию до внедрения" (готовность технологии)

**Принцип:** Чем выше готовность технологии, тем ближе к центру она располагается (меньший радиус).

#### В.2. Факторы готовности

Радиус рассчитывается на основе четырех факторов готовности:

1. **techRead (0-3)** - Технологическая готовность
2. **organRead (0-3)** - Организационная готовность
3. **funcCover (0-3)** - Покрытие функций (автоматический расчет)
4. **trlStage (1-3)** - TRL стадия

#### В.3. Нормализация факторов

Все факторы нормализуются в диапазон [0, 1]:
```
x_techRead = techRead / 3
x_organRead = organRead / 3
x_funcCover = funcCover / 3
x_trlStage = (trlStage - 1) / 2
```

#### В.4. Сводный показатель готовности

Вычисляется как взвешенная сумма нормализованных факторов:
```
z_i = w_techRead * x_techRead + w_organRead * x_organRead +
      w_funcCover * x_funcCover + w_trlStage * x_trlStage
```

**Веса факторов (по умолчанию):**
- techRead: 30%
- organRead: 30%
- funcCover: 20%
- trlStage: 20%

#### В.5. Расчет радиуса

**Формула:**
```
r_i = r_min + (r_max - r_min) * (1 - z_i)
```

где:
- `r_i` - радиус технологии в процентах (0 < r < 100)
- `r_min` - минимальный радиус (по умолчанию 5%)
- `r_max` - максимальный радиус (по умолчанию 95%)
- `z_i` - сводный показатель готовности (0 ≤ z_i ≤ 1)

**Интерпретация:**
- Малый радиус (близко к центру) = высокая готовность к внедрению
- Большой радиус (близко к краю) = низкая готовность к внедрению

#### В.6. Расчет угла

Квадрант определяется на основе направлений цифрового развития технологии.

**Формула расчета угла:**
```javascript
quadrantHash = tech.id * 1000 + quadrantId * 37
angleOffset = (quadrantHash * GOLDEN_ANGLE) % ANGLE_SPAN
theta = startAngle + ANGLE_PAD + angleOffset
```

где:
- `GOLDEN_ANGLE = 137.50776405003785°` - золотой угол для равномерного распределения
- `ANGLE_SPAN = 90 - (ANGLE_PAD * 2)` - доступный угловой диапазон квадранта
- `ANGLE_PAD = 8°` - отступ от границ квадранта
- `startAngle` - начальный угол квадранта (0°, 90°, 180°, 270°)
