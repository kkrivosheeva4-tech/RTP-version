# Матрица ролей и прав

**Статус:** baseline  
**Назначение:** кратко зафиксировать, какой функционал доступен ролям `guest`, `editor`, `owner`, `admin` в текущей версии приложения.

## Как читать таблицу

- `Да` — роль имеет прямой доступ к функционалу.
- `Нет` — доступ отсутствует.
- `Через proposal` — прямого изменения нет, работа выполняется через создание предложения на изменение с последующим согласованием.

## Роли

| Роль | Краткое назначение |
| --- | --- |
| `guest` | Базовый пользователь с доступом на чтение данных радара и связанных справочников. |
| `editor` | Пользователь, который может инициировать изменения через moderation flow, но не публикует их напрямую. |
| `owner` | Владелец доменной области: может напрямую управлять технологиями и согласовывать предложения. |
| `admin` | Администратор с полным доступом к предметной части и административным функциям системы. |

## Ключевые правила доступа

- Все четыре роли имеют доступ на чтение радара и справочников после авторизации.
- `editor` не управляет технологиями напрямую: его рабочий путь — только через `proposal`.
- `owner` сочетает две зоны ответственности: прямое управление технологиями и модерацию предложений.
- `admin` наследует все права `owner` и дополнительно получает доступ к административному контуру системы.

## Примечание по legacy-ролям

В проекте используется role model v2: `guest`, `editor`, `owner`, `admin`. Исторические роли вроде `architect`, `director`, `project_manager`, `analyst`, `viewer` рассматриваются только как слой совместимости и при runtime-нормализации приводятся к ролям v2.
