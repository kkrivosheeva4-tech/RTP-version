# Модель размещения технологий на технологическом радаре

## Обзор

Технологический радар представляет технологии в полярной системе координат, где каждая технология отображается точкой с координатами **(θ, r)**:

- **θ (тета, угол)** — направление, определяемое принадлежностью технологии к функциональному сектору/кластеру
- **r (радиус)** — расстояние от центра, вычисляемое как мера «дистанции до внедрения»

Радиус строго находится в диапазоне **(0, 100)** и никогда не достигает граничных значений 0 или 100.

---

## 1. Вычисление угла (θ)

### Определение сектора

Угол технологии определяется её принадлежностью к одному из четырёх квадрантов радара:

1. **Квадрант 1** (0°–90°): Корпоративное управление и администрация
2. **Квадрант 2** (90°–180°): Основное производство
3. **Квадрант 3** (180°–270°): Производственная поддержка и безопасность
4. **Квадрант 4** (270°–360°): Внешние бизнесы

### Связь с блоками

Каждая технология привязана к одному или нескольким **функциональным блокам** через поле `blocks` или `block`. Каждый блок маппится на квадрант через таблицу соответствия `blockToQuadrant`.

Если технология принадлежит нескольким блокам, связанным с разными квадрантами, она может быть отображена в каждом соответствующем квадранте отдельной точкой.

### Формула угла

```
θ = startAngle + ANGLE_PAD + offset

где:
- startAngle — начальный угол квадранта (0°, 90°, 180°, 270°)
- ANGLE_PAD — отступ от границ квадранта (обычно 8°)
- offset = (ID × GOLDEN_ANGLE) mod ANGLE_SPAN
  - ID — уникальный идентификатор технологии
  - GOLDEN_ANGLE = 137.50776405003785° (золотой угол для равномерного распределения)
  - ANGLE_SPAN = 90° - (2 × ANGLE_PAD) — доступная ширина квадранта
```

**Пример**: Технология с ID=1 в квадранте 2 (startAngle=90°) будет иметь базовый угол около 90° + 8° = 98°, с небольшим смещением на основе ID для избежания наложений.

## 2. Вычисление радиуса (r)

Радиус вычисляется как **«дистанция до внедрения»** на основе математической модели с логистической функцией.

### 2.1 Факторы оценки

Система использует следующие факторы, каждый в своей шкале:

| Фактор                     | Поле данных | Диапазон | Описание                                                       |
| -------------------------- | ----------- | -------- | -------------------------------------------------------------- |
| Технологическая готовность | `techRead`  | 0–3      | Готовность технологии с технической точки зрения               |
| Организационная готовность | `organRead` | 0–3      | Готовность организации к внедрению                             |
| Покрытие функций           | `funcCover` | 0–3      | Насколько полно технология покрывает необходимые функции       |
| TRL стадия                 | `trlStage`  | 1–3      | Стадия технологической готовности (Technology Readiness Level) |

### 2.2 Конвейер вычисления радиуса

#### Шаг 1: Нормализация факторов

Все факторы нормализуются в диапазон **[0, 1]**:

```javascript
x_techRead = techRead / 3; // 0–3 → 0–1
x_organRead = organRead / 3; // 0–3 → 0–1
x_funcCover = funcCover / 3; // 0–3 → 0–1
x_trlStage = (trlStage - 1) / 2; // 1–3 → 0–1
```

#### Шаг 2: Вычисление сводного показателя

Сводный показатель рассчитывается как взвешенная сумма нормализованных факторов с добавлением сдвига (bias):

```
z_i = Σ(w_k × x_ik) + b
```

**Веса факторов (w_k)**:

| Фактор      | Вес   | Направление влияния              |
| ----------- | ----- | -------------------------------- |
| `techRead`  | +0.25 | Положительный (уменьшает радиус) |
| `organRead` | +0.25 | Положительный (уменьшает радиус) |
| `funcCover` | +0.15 | Положительный (уменьшает радиус) |
| `trlStage`  | +0.20 | Положительный (уменьшает радиус) |

**Сдвиг (bias)**: `b = -0.5`

Отрицательный сдвиг делает модель более строгой: при одинаковых оценках технология будет расположена дальше от центра.

**Пример расчёта**:

Технология с оценками:

- `techRead = 3` → `x_techRead = 1.0`
- `organRead = 2` → `x_organRead = 0.67`
- `funcCover = 3` → `x_funcCover = 1.0`
- `trlStage = 3` → `x_trlStage = 1.0`

```
z_i = (0.25 × 1.0) + (0.25 × 0.67) + (0.15 × 1.0) + (0.20 × 1.0) + (-0.5)
   = 0.25 + 0.1675 + 0.15 + 0.20 - 0.5
   = 0.2675
```

#### Шаг 3: Преобразование через логистическую функцию

Логистическая функция преобразует сводный показатель `z_i` в вероятность близости к внедрению:

```
p_i = 1 / (1 + exp(-α × z_i))
```

**Параметр чувствительности**: `α = 4` (рекомендуемый диапазон 3–5)

Логистическая функция обеспечивает:

- Плавный переход между состояниями
- Асимптотическое приближение к границам 0 и 1
- Чувствительность к изменениям факторов в средней зоне

**Пример** (продолжение):

```
p_i = 1 / (1 + exp(-4 × 0.2675))
   = 1 / (1 + exp(-1.07))
   = 1 / (1 + 0.343)
   = 1 / 1.343
   ≈ 0.745
```

Значение `p_i = 0.745` означает высокую степень готовности (74.5%).

#### Шаг 4: Применение правил размещения

Перед вычислением финального радиуса применяются правила размещения технологий:

**Правило 1: Центральный круг**

- Технологии с максимальными оценками (techRead=3, organRead=3, funcCover=3, trlStage=3) **И** большим количеством вендоров (≥ 2) могут располагаться в центральном круге (r < 35)

**Правило 2: Среднее кольцо**

- Если хоть одна оценка меньше 3, технология принудительно размещается в среднем кольце (r ≥ 35)
- Если все оценки максимальные, но вендоров меньше 2, технология также размещается в среднем кольце (r ≥ 35)

**Логика применения**:

```javascript
isMaxScores =
  techRead === 3 && organRead === 3 && funcCover === 3 && trlStage === 3;
hasManyVendors = vendorsCount >= 2;

if (!isMaxScores) {
  // Хоть одна оценка не максимальная → принудительно в среднее кольцо
  p_i = min(p_i, (100 - 35) / 100); // r >= 35
} else if (!hasManyVendors) {
  // Все максимальные, но мало вендоров → в среднее кольцо
  p_i = min(p_i, (100 - 35) / 100); // r >= 35
}
// Если isMaxScores && hasManyVendors, используем расчетный p_i
```

#### Шаг 5: Вычисление радиуса

Радиус вычисляется как инвертированная вероятность:

```
r_i = 100 × (1 - p_i)
```

Затем применяется гарантия минимума для среднего кольца:

```javascript
if (!isMaxScores || !hasManyVendors) {
  if (r_i < 35) r_i = 35;
}
```

**Интерпретация**:

- Центральный круг (r < 35) = максимальные оценки + много вендоров (≥ 2)
- Среднее кольцо (r ≥ 35) = не все максимальные оценки или мало вендоров
- Внешнее кольцо (r > 75) = низкие оценки готовности

**Пример** (технология с максимальными оценками и 5 вендорами):

```
r_i = 100 × (1 - 0.802)
   = 100 × 0.198
   ≈ 19.8
```

Радиус ≈ 19.8 означает, что технология расположена в центральном круге (все оценки максимальные + много вендоров).

**Пример** (технология с techRead=3, organRead=3, funcCover=2, trlStage=3):

```
// funcCover=2 → не все максимальные → принудительно в среднее кольцо
r_i = max(расчетный_радиус, 35)
   = 35 (минимум)
```

#### Шаг 6: Гарантия границ

Для обеспечения строгого диапазона **(0, 100)**, радиус ограничивается:

```javascript
const EPSILON = 0.01;
if (r_i <= 0) r_i = EPSILON; // Минимум 0.01
if (r_i >= 100) r_i = 100 - EPSILON; // Максимум 99.99
```

## 3. Размер технологий на радаре

Размер точки, представляющей технологию на радаре, зависит от количества вендоров, связанных с технологией. Это позволяет визуально оценить экосистему решений для каждой технологии.

### 3.1 Принцип определения размера

Размер технологии определяется количеством вендоров и применяется только на **директорской странице** (RMK-director.html). На обычной странице (RMK.html) все технологии имеют фиксированный размер.

### 3.2 Таблица размеров

Система использует три разреза для определения размера технологии:

| Количество вендоров | Размер (радиус в пикселях) | Визуальная категория |
| ------------------- | -------------------------- | -------------------- |
| 0-1                 | 8                          | Маленький            |
| 2-3                 | 14                         | Средний              |
| 4+                  | 20                         | Большой              |

### 3.3 Формула вычисления размера

```javascript
if (vendorCount <= 1) {
  // Маленький: 0-1 вендор
  size = 8;
} else if (vendorCount === 2 || vendorCount === 3) {
  // Средний: 2-3 вендора
  size = 14;
} else {
  // Большой: 4+ вендоров
  size = 20;
}
```

### 3.4 Интерпретация размера

- **Маленький размер (8px)**: Технология с 0-1 вендором — ограниченный выбор решений
- **Средний размер (14px)**: Технология с 2-3 вендорами — умеренный выбор решений
- **Большой размер (20px)**: Технология с 4+ вендорами — широкий выбор решений, развитая экосистема

### 3.5 Связь с принятием решений

Размер технологии помогает директорам и руководителям проектов:

- **Быстро оценить зрелость рынка**: Большие точки указывают на технологии с развитой экосистемой вендоров
- **Выявить риски**: Маленькие точки могут указывать на технологии с ограниченным выбором решений
- **Приоритизировать исследования**: Технологии с большим количеством вендоров могут быть более безопасным выбором

**Примечание**: Размер применяется только к радиусу/размеру фигуры технологии (круг, квадрат, треугольник, звезда). Форма технологии определяется типом технологии (`techType`), а не количеством вендоров.

## 4. Маппинг радиуса на координаты SVG

После вычисления радиуса в процентах (0–100), он масштабируется к реальным координатам SVG:

```
radius_pixels = PAD + (r_i / 100) × availableRadius

где:
- PAD — отступ от центра (обычно 30px)
- availableRadius — доступный радиус с учётом максимального кольца
  = (RINGS.length × RADIUS_STEP) - PAD
- RADIUS_STEP — шаг между кольцами (обычно 140px)
```

**Итоговые декартовы координаты**:

```
x = CENTER_X + radius_pixels × cos(θ - 90°)
y = CENTER_Y + radius_pixels × sin(θ - 90°)
```

## 5. Интерпретация визуального представления

### Расстояние от центра

| Позиция          | Радиус (r_i) | Условия размещения                        | Интерпретация                                                       |
| ---------------- | ------------ | ----------------------------------------- | ------------------------------------------------------------------- |
| Центральный круг | 0–35         | Все оценки = 3 **И** вендоров ≥ 2         | Технология готова к внедрению, максимальные оценки и много вендоров |
| Средняя зона     | 35–50        | Хоть одна оценка < 3 **ИЛИ** вендоров < 2 | Технология в процессе подготовки, есть области для улучшения        |
| Средняя-внешняя  | 50–75        | Низкие оценки готовности                  | Технология требует дополнительной подготовки, низкие оценки         |
| Внешняя зона     | 75–100       | Очень низкие оценки готовности            | Технология далека от внедрения, требует значительных усилий         |

### Влияние отдельных факторов

**Факторы (уменьшают радиус, приближают к центру)**:

- Высокий `techRead` (3) → уменьшает радиус на ~8.3 единиц при весе 0.25
- Высокий `organRead` (3) → уменьшает радиус на ~8.3 единиц
- Высокий `funcCover` (3) → уменьшает радиус на ~5 единиц
- Высокий `trlStage` (3) → уменьшает радиус на ~6.7 единиц

**Комбинированный эффект**:

Технология с максимальными оценками (techRead=3, organRead=3, funcCover=3, trlStage=3) будет иметь радиус около **15–20** (близко к центру).

Технология с минимальными оценками (techRead=0, organRead=0, funcCover=0, trlStage=1) будет иметь радиус около **75–85** (далеко от центра).

## 6. Примеры расчётов

### Пример 1: Высокая готовность

**Данные технологии**:

```json
{
  "id": 1,
  "name": "MES добычи",
  "techRead": 3,
  "organRead": 2,
  "funcCover": 3,
  "trlStage": 3,
  "blocks": [1, 20]
}
```

**Расчёт**:

1. Нормализация: `x_techRead=1.0`, `x_organRead=0.67`, `x_funcCover=1.0`, `x_trlStage=1.0`
2. Сводный показатель: `z_i = 0.25 + 0.1675 + 0.15 + 0.20 - 0.5 = 0.2675`
3. Логистическая функция: `p_i = 1 / (1 + exp(-1.07)) ≈ 0.745`
4. Радиус: `r_i = 100 × (1 - 0.745) = 25.5`
5. Угол: определяется квадрантом блока 1 (обычно квадрант 1 или 2)

**Результат**: Технология расположена близко к центру (r ≈ 25.5), что отражает высокую готовность.

### Пример 2: Низкая готовность

**Данные технологии**:

```json
{
  "id": 8,
  "name": "AI-планирование буровзрывных работ",
  "techRead": 1,
  "organRead": 2,
  "funcCover": 3,
  "trlStage": 1
}
```

**Расчёт**:

1. Нормализация: `x_techRead=0.33`, `x_organRead=0.67`, `x_funcCover=1.0`, `x_trlStage=0.0`
2. Сводный показатель: `z_i = 0.083 + 0.1675 + 0.15 + 0.0 - 0.5 = -0.10`
3. Логистическая функция: `p_i = 1 / (1 + exp(0.4)) ≈ 0.401`
4. Радиус: `r_i = 100 × (1 - 0.401) = 59.9`

**Результат**: Технология расположена дальше от центра (r ≈ 59.9), что отражает низкую технологическую готовность и низкую TRL стадию.

## 7. Динамическое обновление

При изменении оценок технологии (например, через форму редактирования) радиус автоматически пересчитывается:

1. Изменение факторов → новый `z_i`
2. Новый `z_i` → новый `p_i` через логистическую функцию
3. Новый `p_i` → новый `r_i`
4. Новый `r_i` → новое положение точки на радаре

Перемещение точки может быть анимировано для плавного визуального перехода.

## 8. Связь с бизнес-логикой

### Оценки готовности → Позиция на радаре

Система оценки технологий напрямую влияет на визуальное представление:

| Оценка                          | Влияние на радиус  | Визуальный эффект      |
| ------------------------------- | ------------------ | ---------------------- |
| Высокие `techRead`, `organRead` | Уменьшает радиус   | Точка ближе к центру   |
| Низкие `techRead`, `organRead`  | Увеличивает радиус | Точка дальше от центра |
| Высокий `funcCover`             | Уменьшает радиус   | Точка ближе к центру   |
| Высокий `trlStage`              | Уменьшает радиус   | Точка ближе к центру   |

### Принятие решений

Директора и РП могут использовать визуальное представление для:

- **Быстрой оценки**: Технологии ближе к центру требуют меньше ресурсов для внедрения
- **Приоритизации**: Фокус на технологиях с оптимальным балансом готовности и стоимости
- **Мониторинга прогресса**: Изменение позиции при обновлении оценок показывает динамику подготовки

Основная функция реализована в модуле `src/js/modules/radar/positioning.js`:

```javascript
function calculateRadarPosition(tech) {
  // ... вычисление угла и радиуса
```

Функция вызывается при:

- Первичной загрузке данных

## Заключение

Модель размещения технологий на радаре обеспечивает:

1. **Объективность**: Позиция вычисляется автоматически на основе количественных оценок
2. **Прозрачность**: Понятная математическая модель, связывающая оценки с визуальным представлением
3. **Гибкость**: Настраиваемые параметры позволяют адаптировать модель под потребности организации
4. **Динамичность**: Автоматическое обновление позиций при изменении данных

Модель служит инструментом визуализации для поддержки принятия решений в области технологического развития.
