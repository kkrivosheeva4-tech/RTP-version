# План исправлений модальных окон и индикаторов на странице radar.html

Документ описывает выявленные проблемы и пошаговый план их решения.

---

## 1. Индикаторы сохранения в модальных окнах добавления и редактирования технологии

### Проблема
В модальных окнах добавления и редактирования технологии в разметке присутствуют индикаторы загрузки (`#addTechLoadingIndicator`, `#editTechLoadingIndicator`), но при тестировании они не отображаются.

### Причины (по результатам анализа кода)
- **Разметка и стили:** Индикаторы заданы в `radar.html` (строки ~762–766 и ~1109–1113), стили в `RMK.css`: `.form-loading-indicator` по умолчанию имеет `display: none`, при классе `.active` — `display: flex`. Позиционирование: `position: absolute; top: 50%; left: 50%; transform: translate(-50%, -50%);` относительно модального окна.
- **Логика в JS:** В `form-management.js` перед отправкой формы вызывается `loadingIndicator.classList.add('active')`, после сохранения или ошибки — `loadingIndicator.classList.remove('active')`.
- **Вероятная причина невидимости:** сохранение выполняется синхронно (localStorage), поэтому индикатор включается и сразу выключается, и пользователь его не успевает увидеть. Дополнительно возможно перекрытие индикатора другими элементами или отсутствие у модалки `position: relative` для корректного позиционирования.

### Шаги решения (до перехода на индикацию в кнопке — см. п. 5)
1. Убедиться, что у контейнера модального окна (`#addTechPanel`, `#editTechPanel`) задано `position: relative` (или что контекст позиционирования для `.form-loading-indicator` корректен).
2. Проверить порядок слоёв (z-index): у `.form-loading-indicator` задан `z-index: 10000`; убедиться, что он выше, чем у `.modal-body` и `.form-actions`.
3. При необходимости добавить минимальное время показа индикатора (например, 300–500 ms) после `classList.add('active')`, чтобы он гарантированно был виден даже при быстром сохранении.

**Примечание:** В рамках предложения из п. 5 отдельные блоки индикаторов можно убрать и заменить индикацией «загрузка» в самой кнопке «Добавить»/«Сохранить».

## 2. Дублирование загрузки вкладок предприятий в модальных окнах

### Проблема
В модальных окнах добавления и редактирования технологии в начале и в конце упоминается загрузка/размещение вкладок предприятий — нужно понять, это дублирование или разная функциональность.

### Текущая структура
- **В начале (в зоне вкладок):**
  - `#enterpriseTabsContainer` / `#editEnterpriseTabsContainer` — контейнеры для **кнопок вкладок** (табы) предприятий.
  - Комментарии в HTML: «Динамические вкладки предприятий будут добавляться здесь».
- **В конце (в теле формы):**
  - `#enterpriseTabsContent` / `#editEnterpriseTabsContent` — контейнеры для **содержимого** каждой вкладки (контент по предприятиям).

### Вывод
Это не дублирование логики, а два разных контейнера:
- первый — для табов (кнопок переключения),
- второй — для контента выбранной вкладки.

Заполнение выполняют:
- для добавления: `TechTabsManager` (`tech-tabs-manager.js`) — `enterpriseTabsContainer` и `enterpriseTabsContent`;
- для редактирования: `EditTechTabsManager` (`edit-tech-tabs-manager.js`) — `editEnterpriseTabsContainer` и `editEnterpriseTabsContent`.

### Рекомендации
1. Оставить оба контейнера; при необходимости в HTML можно уточнить комментарии, например: «Вкладки предприятий (кнопки)» и «Контент вкладок предприятий».
2. Если в коде есть два отдельных вызова «загрузки вкладок» с одинаковой логикой в одном и том же модуле — вынести общую часть в одну функцию и вызывать её для табов и для контента, чтобы избежать дублирования кода (без изменения поведения).

## 3. Список направлений в окне «Добавить функциональный блок» из JSON

### Проблема
В модальном окне добавления функционального блока (`#addBlockPanel`) список «Направления» захардкожен в HTML (радар.html, строки ~577–584): четыре `<li>` с фиксированными названиями направлений. Требуется загружать направления из `src/data/ru/directionToQuadrant.json`.

### Формат JSON
`directionToQuadrant.json` — объект, ключи которого являются названиями направлений:
```json
{
  "Единый центр данных (Data Hub)": [1],
  "Искусственный интеллект во всех процессах": [2],
  "Автономные комплексы на всех предприятиях": [3],
  "Надёжная и безопасная цифровая инфраструктура": [4]
}
```

### Шаги решения
1. **HTML:** В `#addBlockPanel` у селекта с `data-field="blockSector"` оставить контейнер списка пустым: `<ul class="select-options"></ul>` — без статических `<li>`. Скрытое поле `#blockSector` оставить.
2. **Загрузка данных:** Направления уже подгружаются в `data-loader.js` в переменную `window.directionToQuadrant`. При открытии модального окна «Добавить функциональный блок» (или при инициализации страницы/модалок) заполнять список направлений из `Object.keys(window.directionToQuadrant)` (или из того же источника, что используется для фильтров/форм технологий, если он уже строится из этого JSON).
3. **Место инициализации:** Либо в `modals.js` при открытии `addBlockPanel` (по аналогии с инициализацией других модальных селектов), либо в `form-management.js` в обработчике открытия формы добавления блока. Добавить функцию вида: получить направления из `window.directionToQuadrant`, сформировать `<li data-value="...">...</li>` для каждого названия и вставить в `.custom-select-modal[data-field="blockSector"] .select-options`.
4. **Сброс при закрытии:** При закрытии модалки сброс селекта уже выполняется в `modals.js` (hideModal для addBlockPanel). Убедиться, что при следующем открытии список снова строится из актуального `directionToQuadrant` (при необходимости вызывать заполнение при каждом открытии).
5. **Единообразие:** Проверить, что порядок направлений в селекте совпадает с логикой радара (при необходимости сортировать по значению квадранта из JSON).

## 4. Модальное окно загрузки отчёта не отображается

### Проблема
В разметке после модального окна «Отчеты» (экспорт) присутствует модальное окно загрузки отчёта `#reportLoadingModal` (радар.html, строки ~1466–1494), но при экспорте оно не показывается.

### Причина
- Модуль `report-status.js` реализует `showReportLoading()`, `showReportSuccess()`, `showReportError()` и вешает их на `window`. Эти функции вызывают `window.showModal('reportLoadingModal')` и управляют содержимым (спиннер, успех, ошибка).
- В текущем потоке экспорта PDF (`export.js`, функция `performPdfExport`) используется только `LoadingManager.show('Генерация PDF отчета...')` и `LoadingManager.hide(loaderId)`. Вызовы `showReportLoading` / `showReportSuccess` / `showReportError` нигде не выполняются, поэтому модальное окно загрузки отчёта не открывается.

### Дополнительно
- В `rmk-inline-styles.css` для `#reportLoadingModal` задано `display: none`. При вызове `showModal('reportLoadingModal')` в `modals.js` выставляется `panel.style.display = 'block'` и добавляется класс `open`, поэтому при вызове функции окно должно отображаться.
- Класс `hidden` у `#reportLoadingModal` в HTML при необходимости можно удалить или учитывать в showModal, чтобы не конфликтовать с отображением.

### Шаги решения
1. **Подключить модальное окно к экспорту PDF:** В `export.js` в начале асинхронной операции экспорта (в `performPdfExport` или в обработчике кнопки «Экспортировать» перед вызовом экспорта) вызывать `window.showReportLoading()` вместо или перед `LoadingManager.show(...)`. Решить, что использовать как основной индикатор: только модальное окно отчёта или только LoadingManager. Рекомендация: использовать модальное окно отчёта для сценария «Экспорт отчёта», чтобы соответствовать разметке и ожиданиям пользователя.
2. **Успех:** После успешной генерации PDF вызывать `window.showReportSuccess()` и убрать показ LoadingManager для этого сценария (или вызывать после него hide).
3. **Ошибка:** В блоке `catch` вызывать `window.showReportError(error.message)` (или аналог с текстом ошибки) и при необходимости скрывать LoadingManager.
4. **Стили и доступность:** Убедиться, что при открытии модального окна загрузки отчёта класс `hidden` не оставляет `display: none` с более высокой специфичностью. При необходимости в `showModal` для `reportLoadingModal` принудительно задавать отображение.
5. **Порядок загрузки модулей:** `report-status.js` и модуль, задающий `showModal`/`hideModal`, должны быть загружены до вызова экспорта (уже учтено в app-init и порядке скриптов).

## 5. Предложение: индикация загрузки в кнопках «Добавить» / «Сохранить» / «Экспортировать»

### Идея
Вместо (или вместе с) отдельных блоков индикации в модальных окнах при клике на «Добавить», «Сохранить», «Экспортировать» менять иконку кнопки на единый индикатор загрузки — анимированную SVG-иконку «загрузка» (линия/дуга, движущаяся по кругу), как при загрузке данных при переходе на страницу radar.html (индикатор LoadingManager или аналогичный визуал).

### Что сделать
1. **Единая SVG-иконка загрузки (спиннер):**  
   Подготовить небольшую SVG-иконку (например, круг с обводкой и анимацией вращения или «бегущая» дуга), визуально совпадающую со стилем загрузки на странице радара. Разместить её в разметке или вставлять через JS (например, как в `loading.js` используется `.spinner-ring` в common.css). Вариант: общий класс, например `.btn-loading-spinner`, со стилями из текущего спиннера страницы (например, из `.spinner-ring` в common.css или из стилей модального окна загрузки отчёта).
2. **Поведение кнопок:**
   - **Добавить технологию:** Кнопка `#submitAddTech` (form="addTechForm"). По клику (до отправки формы): отключить кнопку, скрыть обычную иконку и текст «Добавить технологию», показать в кнопке спиннер (и при желании текст «Сохранение…»). После завершения (успех/ошибка) вернуть исходное состояние.
   - **Сохранить (редактирование):** Кнопка submit формы `#editTechForm`. Аналогично: при отправке показать в кнопке спиннер, после ответа — вернуть иконку и текст «Сохранить».
   - **Экспортировать:** Кнопка `#exportPdfConfirm`. По клику показать в кнопке спиннер (и при желании текст «Экспорт…»), по завершении генерации PDF вернуть исходный вид.
3. **Реализация в коде:**
   - В каждой из кнопок предусмотреть контейнер для иконки (например, обёртка с классом `.btn-icon-wrap`) и при необходимости контейнер для спиннера. По умолчанию спиннер скрыт (например, `display: none` или класс `hidden`). При начале загрузки: добавить кнопке класс, например `loading`, скрыть обычную иконку, показать спиннер; после завершения — убрать класс и вернуть иконку.
   - Стили: описать в CSS (например, в RMK.css или common.css) состояние `.btn-primary.loading .btn-icon { display: none }`, `.btn-primary.loading .btn-loading-spinner { display: inline-block }` и анимацию спиннера. Размер спиннера подобрать под высоту кнопки (например, 16–20px).
4. **Совместимость с текущими индикаторами:**
   - Для добавления/редактирования технологии: можно оставить существующие `#addTechLoadingIndicator` и `#editTechLoadingIndicator` скрытыми или удалить их из разметки и убрать вызовы `classList.add/remove('active')` в `form-management.js`, чтобы единственной индикацией было состояние кнопки.
   - Для экспорта: после внедрения п. 4 модальное окно загрузки отчёта может оставаться для полноэкранного сообщения «Загрузка…» / «Успех» / «Ошибка»; при этом кнопка «Экспортировать» может дополнительно показывать спиннер в себе для немедленной обратной связи.
5. **Доступность:** В состоянии загрузки устанавливать кнопке `disabled` и `aria-busy="true"`, при необходимости менять текст для скринридеров (`aria-live` или обновление подписи).

### Файлы для правок
- **HTML:** `radar.html` — при необходимости добавить в кнопки обёртки/спан для спиннера или оставить разметку для вставки спиннера через JS.
- **CSS:** `RMK.css` или `common.css` — классы для состояния кнопки `.loading` и анимация спиннера в кнопке.
- **JS:** `form-management.js` — включение/выключение состояния загрузки для кнопок добавления и сохранения; `export.js` — для кнопки «Экспортировать».

После выполнения плана стоит провести проверку: добавление технологии, редактирование, добавление функционального блока, экспорт PDF — во всех случаях индикация должна быть заметной и предсказуемой.
